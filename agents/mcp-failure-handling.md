# MCP 書き込み障害対処エージェント

MCP（Notion・Craft 等ホスト型コネクタ）への書き込みが `requires approval` で失敗したときの対処手順。
検知したら指示がなくても自動実行する（2026-06 佐藤指示）。

---

## 1. 書き込みが `requires approval` で止まった場合

原因はコネクタ側の書き込み同意／MCP セッションの失効。settings.json に当該ツールが入っていても起きる＝**settings では直らない／権限の問題ではない**。読み取りは通るのに書き込みだけ落ちるのが典型。

### 対処手順

1. **リトライを連打しない。** 一度 wedged になると同一セッションでは復帰しにくい。再試行は最大1回、駄目なら止める。
2. **読み取りが生きているか1本だけ確認**（例：notion-fetch を1回）。読めて書けない＝書き込み同意の失効、と切り分ける。
3. **すぐユーザーに通知する。** 「Notion 書き込みが承認待ちで止まりました。これはコネクタ側のエラーで、**チャットに承認ダイアログは出ません**。新セッションでの再開を推奨します」と、原因の一言＋推奨アクションを明示する。
4. **成果物と『保存仕様』を退避する。** 書き込みが生きている作業場（通常 Craft。Craft も落ちていれば全文をチャットに最終ブロックで出力）に、(a) 確定済み成果物の全文、(b) 本来の保存先（Notion の collection／page・各プロパティ等、再現に必要な粒度）を残す。退避先は Craft `work > 10_Recruitment > _引き継ぎ｜未保存タスク`（folder ID: `A50317A7-DDAA-42D2-A0DA-6093E8F60AEC` 直下。無ければ作成）に**日時付きで1エントリ追記**する。
5. **引き継ぎプロンプトをチャットに出力する。** Craft 退避（手順4）と**併せて**、新セッションにそのまま貼れる自己完結プロンプトをコードブロックで出力する。Craft が落ちていてもチャット上にプロンプトが残るので、ユーザーはコピペだけで再開できる。テンプレートは §1.1 に従う。
6. **再開の案内。** 「下のプロンプトを新しいセッションに貼ってください」と一言添える。

### 1.1 引き継ぎプロンプトのテンプレート

障害検知時に以下を**コードブロック（```）で出力**する。`{...}` は実際の値に埋めること。空欄のセクションは省略してよい。

````
```
# MCP 書き込み障害からの引き継ぎ

## 状況
- 前セッションで {対象コネクタ名（Notion / Craft 等）} への書き込みが `requires approval` で停止
- 読み取りは正常 / 読み取りも停止（該当する方を残す）
- 発生日時：{JST}

## 元タスク
{何をしようとしていたかの1-2文の要約}

## 完了済みの作業
{箇条書き。例：候補者プロファイル作成済み、推薦文の下書き完了 等}

## 未保存の成果物

### 成果物①：{成果物の名前}
保存先：{Notion collection / Craft document 等の具体的な保存先。ID を含む}
プロパティ：{Notion の場合はプロパティ名と値の一覧}

本文：
{成果物の全文をここに貼る}

### 成果物②：...（複数あれば繰り返す）

## Craft 退避先（書けた場合）
- Craft `_引き継ぎ｜未保存タスク` folder ID: `A50317A7-DDAA-42D2-A0DA-6093E8F60AEC`
- エントリタイトル：{退避時に付けたタイトル}

## やってほしいこと
1. 上記の未保存成果物を保存先に書き込む
2. {その他、元タスクで残っていた作業があれば}
3. 完了したら Craft 退避エントリに「✅ 保存済み（日時）」を追記
```
````

**テンプレート運用ルール：**

- 成果物の本文は**省略せず全文**を含める（新セッションが Craft 退避を読めない場合の保険）
- Notion のプロパティは `Title` / `Status` / relation 等、再現に必要なものを全て列挙する
- 複数の成果物がある場合は成果物ごとに分ける
- Craft 退避が成功していれば退避先セクションも埋める。失敗していれば「Craft 退避：失敗（本プロンプトが唯一の退避）」と明記する

### 新セッション側の振る舞い

「引き継ぎから再開」「前回の未保存分」等が言及されたら、**まず貼られたプロンプトの内容を優先的に使う**。プロンプトが無い場合は Craft `_引き継ぎ｜未保存タスク` の最新エントリを読む。退避された成果物と保存仕様どおりに**書き込みのみ**を再実行する。完了したら Craft 退避エントリに「✅ 保存済み（日時）」を追記する。

## 2. 特定の読み取りツールだけが `requires approval` で落ちる場合

Google Drive で `search_files` だけが落ち、`list_recent_files` / `get_file_metadata` / `read_file_content` は通る事象を確認（2026-06）。クエリ種別に依らず `search_files` 単体が落ちる＝権限ではなくツール別同意／スコープ不足。

### 切り分けと対処

1. **コネクタ全体の死活を1本で切り分ける。** 同じサーバーの別の読み取り（Drive なら `list_recent_files` を1回）を叩く。通れば「コネクタは生きており、そのツールだけ同意切れ」と確定する。
2. **恒久対策（要・佐藤操作）：** Claude のコネクタ設定で当該コネクタ（Google Drive 等）を**再接続・再認可**してスコープ同意をやり直す。Web セッションには承認ダイアログが出ないため、設定/アプリ側での再接続が必要。docs：https://code.claude.com/docs/en/claude-code-on-the-web
3. **当座の回避策（Claude 側で即適用可・`search_files` 代替）：** 候補者/クライアントの Drive 探索（`agents/candidate-profile.md` §3・`agents/client-profile.md` §4）は `search_files` 前提だが、落ちている間は次で代替する。(a) 佐藤が直近保存したやりとり/レジュメは `list_recent_files`（recency 順）に出るので拾える、(b) folder/fileId が既知なら `get_file_metadata` + `read_file_content` で直接取得、(c) フォルダ横断や過去ファイルの全文検索は完全代替できないため、その範囲は「未取得（Drive search 承認エラー）」と明記し、再接続後の同期で追補する。リトライ連打はしない。
