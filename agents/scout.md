# スカウト文作成エージェント

候補者へ送る一次接触メッセージの作成ガイド。
**Notion スカウト DB**（`collection://2597d017-b6a0-801b-8185-000ba4b9661e`）に保管されている直近のテンプレート（2026 年）の文体・構造に準拠する。

---

## 1. 作成プロセス

### Step 0：ペルソナ設定（最初に決める）
構成・訴求・件名・CTA をペルソナに合わせて変えるため、書き始める前に候補者を **3ペルソナ（P1/P2/P3、セクション2）** のどれかに分類する。年齢は代理指標で、実際は次の3つで見分ける：

- **フェーズ**：まだ上を目指す（上り志向）か／既に地位があり意義で動く（充足）か
- **市場での引き合い**：売れ線（スカウト過多で埋もれやすい）か／ニッチか
- **動機の重心**：成長・年収・裁量 か／意義・問い・座組 か

該当ポジションの **scout-kit に `ターゲット人物像.md` があれば、まずそれを読む**（`agents/scout-kit.md` §4.0）。Tier1/2/3 ごとの想定年齢層・訴求軸が定義されているので、勘で決めず Tier をペルソナ（P1/P2/P3）にマッピングして構成・件名・CTA を決める。無ければ下記3軸で判断する。

LinkedIn の志向や過去のキャリア選択で年齢バンドは上書きしてよい。選んだペルソナはスカウト DB にも記録する（送信時にペルソナ＝年代バンドを残すと、後で層別返信率が出せる。返信者しか年齢が載らない現状の生存者バイアスを将来解消するため）。

### Step 1：前提情報の確認
書き始める前に以下をユーザーに確認する：

- [ ] 想定ペルソナ（P1 / P2 / P3）
- [ ] 候補者の宛名形式（`{NAME}様` / `{姓}様` / `[候補者名]様` のどれか）
- [ ] 媒体（Bizreach / LinkedIn / dodaX / eight / リクナビHRTech / YOUTRUST / SF）
- [ ] 担当者名（**佐藤** か **古市** か）：自分が直接送るか、紹介者と担当が分かれているか
- [ ] 企業名・業界の出し方（完全に伏せる／属性のみ出す／一部特定可）
- [ ] ポジションの本質を表すフック（一言）

### Step 2：訴求3軸の整理
- 会社概要（事業ステージ・市場ポジション・成長率）
- ポジションの魅力（3〜4点）
- 報酬イメージ

### Step 3：本文作成
下記の構造に従って書く。

### Step 4：保存・まとめ（出力工程・必須）
作成した文面は「作って終わり」にせず、**(1) スカウトDB に媒体ごとのレコード**、**(2) ポジションページ直下の「まとめページ」**の2か所に保存する。詳細手順はセクション9。

## 2. ペルソナ別の構成・訴求設計（最重要）

返信率は層によって効くレバーが違う。軸は「経歴への言及あり／なし」ではなく、**何をパーソナライズするか・メッセージの重心・トーン**である。称賛型の経歴言及はシニアほど効きにくいため、評価する文体ではなく口説く文体に倒す。既存テンプレ（セクション3）の短文版＝若手型、長文版＝シニア型に対応する。

### P1｜売れ線・上り（20代後半〜30代前半が中心）
- **重心**：あなたのフィット＋伸びしろ
- **件名**：年収・成長・領域の数字で開封を取る
- **構成**：実績引用フック → 一行自己紹介 → なぜ今この役割（成長・裁量・upside）→ 報酬 → 低ハードルCTA（「15分」「情報提供だけでも」）
- **形式**：短文版・箇条書き
- **効くパーソナライズ**：具体的な実績を引用する（homework した誠実さの証明になり刺さる）

### P2｜中堅・マネジメント（30代後半〜40代前半）
- **重心**：役割の広がり×フィット
- **件名**：役割の広さ・裁量・事業インパクト
- **構成**：ハイブリッド。実績への言及は「称賛」ではなく「だからこの裁量をお任せしたい」に接続する
- **形式**：短〜中文
- **効くパーソナライズ**：実績を、任せたい裁量・役割の広がりへ接続する

### P3｜シニア・エグゼク（40代後半〜、CxO・部長・専門責任者）
- **重心**：状況の希少性・意義・本気度
- **件名**：案件の本質・意義・希少性（年収より問い）
- **構成**：長文版。状況・課題の描写から入る → 希少性（これができる方は限られる、だからご連絡した）→ 役割の射程 → 報酬は柔軟に控えめ → 壁打ち・情報共有型CTA＋秘匿性の担保
- **形式**：長文版・地の文
- **効くパーソナライズ**：過去実績の称賛ではなく、前を向いた仮説と「希少な解」としての位置づけ。秘匿性・本気度（実在する具体的な依頼であること）自体が返信動機になる

**全ペルソナ共通の注意：** 一斉送信臭はどの層にも致命的（シニアほど嫌われる）。パーソナライズはゼロにしない。変えるのは経歴への言及量ではなく、称賛型を減らし、希少性・状況・目線合わせに振ること。

## 3. スカウト文の構造（直近テンプレ）

```
{宛名}様
突然のご連絡失礼いたします。<br>{注力領域の表現}を中心にご支援しております、Pole&Lineの{担当者}と申します。

{連絡理由：{宛名}様のご経歴を拝見し、{ポジションの本質的価値}にご関心がおありであればと思い、ご連絡いたしました。}<br>ご転職の温度感が高くない可能性もあるかと存じますが、以下ご一読いただけますと幸いです。

■会社概要<br>{1〜2文で：上場ステータス・事業概要・市場ポジション・成長率・直近の動き}

■ポジションの魅力<br>・{魅力1}<br>・{魅力2}<br>・{魅力3}<br>・{魅力4}

■報酬<br>年収 {下限}〜{上限}万円{＋ストックオプション等}<br>※ご経験・ご志向に応じて柔軟に検討可能

{結びのフック：「もし少しでもご関心があれば」または、ポジションの本質を一言で表現した印象的な締め}
ご返信をお待ちしております。

Pole&Line<br>{担当者}
```

### バリエーション：長文版（経営ポジション・社会的意義の大きい案件）

短い箇条書きではなく、地の文中心で構造を語る形式も使う：

```
{宛名}様
ご覧いただきありがとうございます。<br>ハイレイヤー案件を中心にご支援しております、Pole&Lineの{担当者}です。

今回は、{領域}において、{ポジションの本質}について、ご相談したくご連絡いたしました。

{現状の事業状況・課題を2〜3段落で描写。
箇条書きは・を並べて要素を示す。
「単なる○○ではなく、〜に近い難易度の高いチャレンジ」のような表現を入れる。}

ポジションとしては、{役割の幅}まで一気通貫で担っていただく想定で、将来的には{将来像}の可能性もある役割です。

なお、{業界経験}は必須ではなく、むしろ、
・{歓迎経験1}
・{歓迎経験2}
・{歓迎経験3}
などをお持ちの方にこそ、ご活躍いただける環境ではないかと感じています。

ご転職の温度感は高くないかもしれませんが、もし少しでもご関心お持ちいただけそうでしたら、詳細は担当の者より、ご説明させていただきます。
ご返信お待ちしております。

Pole&Line<br>{担当者}
```

## 4. 守るべきルール（直近テンプレに準拠）

### 書き出し
- **「突然のご連絡失礼いたします」** が直近の主流。
- 返信スレッドや2回目接触なら「ご覧いただきありがとうございます」。
- 自己紹介は `{領域}を中心にご支援しております、Pole&Lineの{担当者}と申します。` がフォーマット。

### 連絡理由
- 「ご経歴を拝見し、〜にご関心がおありであればと思い、ご連絡いたしました。」
- 「ぜひ一度お話できればと思いご連絡いたしました。」
- 「〜について、ご相談したくご連絡いたしました。」

### 転職温度感への配慮（必須）
以下のいずれかで「転職活動中でなくても OK」感を出す：
- 「ご転職の温度感が高くない可能性もあるかと存じますが」
- 「ご転職のご意向が強くない可能性も承知しておりますが」
- 「ご転職の温度感は高くないかもしれませんが」

### 企業情報の出し方
- **完全に伏せない**のが直近のスタイル。属性レベルで特定できる程度の情報を出して訴求力を上げる：
  - 「プライム上場のプラットフォーマー」
  - 「東京都全体の行政DXを担う組織」
  - 「クラフトビール領域で複数事業を横断的に統括」
- 企業名は出さないが、業界・ステージ・上場区分は出していい。
- ただし、クライアントから「完全秘匿」と指示があれば従う。

### 報酬の書き方
- レンジ + 「※ご経験・ご志向に応じて柔軟に検討可能」を必ず添える
- ストックオプション・インセンティブの有無は明示する
- 「（成果に応じて随時付与）」など条件があれば括弧書きで補足

### 結びのフック
ポジションの本質を一言で表現する。テンプレ化せず案件ごとに考える：
- 「キャリアの最終章で、社会基盤そのものを設計する」（CISO・公共系）
- 「リアル産業をどう再設計するか」（既存産業DX系）
- 「クラフトビール産業のインフラを作るというテーマ」（産業構築系）
- 「守るだけでなく事業の当事者として法務キャリアを築く」（職種転換系）

### 媒体別の改行
- **Bizreach / LinkedIn / dodaX 等の媒体内で送る場合** → `<br>` タグを使う（実テンプレに準拠）
- メール（Gmail）でそのまま送る場合 → 通常の改行で OK
- ユーザーから指定なければデフォルトは `<br>` 形式

### 署名（媒体別・2026-06 佐藤指示）
- **LinkedIn は署名を付けない**（プラットフォームが初回メッセージに自動署名するため。手で付けると二重になる）。
- **ビズリーチ・dodaX 等その他の媒体は署名を付ける**：末尾に `Pole&Line<br>{担当者}`（例：`Pole&Line<br>佐藤`。ビズリーチを古市アカウントで送る場合は `Pole&Line<br>古市`）。
- 媒体ごとに送信アカウントが分かれる場合（例：ビズリーチ＝古市／LinkedIn・dodaX＝佐藤）は、各媒体トグルの**自己紹介・署名をその送信者名に合わせる**。古市など佐藤以外が送る媒体では、本文の結びに「詳細は担当の佐藤よりご説明させていただきます」を入れて担当を明示する。
- ※ `agents/candidate-mail.md` §11 の「LinkedIn 初回は自動署名なので付けない／2通目以降は軽い署名」は候補者メール文脈の話。スカウト一次接触でも **LinkedIn のみ署名なし**で整合する（他媒体は上記のとおり署名あり）。

### NG
- 候補者の現職を批判・推測しない
- 「絶対にマッチします」など断定的な売り込みはしない
- 報酬を未確定のまま盛らない
- 過度な伏字（「ある〇〇企業」だけで終わらせない）：訴求力が落ちる

## 5. タイトルの作り方

スカウト DB の `タイトル` プロパティに保存される。実例：
- `【プライム上場・M&A海外展開加速中】事業の構想初期から関わる企業法務のご提案`
- `年収〜2500万円程度／ヘルスケア×リアル産業DX【事業統括】`
- `東京都全体のサイバーセキュリティ戦略統括ポジションをご提案`
- `クラフトビール市場変革を担う経営ポジションのご相談`

### タイトル作成のコツ
- 【】で訴求軸を冒頭に置く（上場・年収・領域など）
- 「ご提案」「ご相談」など柔らかい語尾で締める
- 数字（年収・成長率）が出せれば入れる
- ポジション名そのままではなく、本質を表現する

## 9. 保存・まとめ工程（出力先：Notion）

スカウト文を作成したら、必ず次の2か所に保存する。これが Step 4＝「作って終わり」にしないための出力工程。

### 9-1. スカウトDB にレコード作成（媒体ごとに1枚）

送信/返信を追跡する正本。`mcp__Notion__notion-create-pages` で、parent＝`data_source_id: 2597d017-b6a0-801b-8185-000ba4b9661e` に、**媒体ごと1レコード**作成する。主要プロパティ：

- `識別ID`（title）：`［vN］{ポジション名}-{社コード}`（例：`［v2］新規事業開発（役員直下）-GFT`）
- `タイトル`（text）：件名
- `DB`（multi_select）：媒体。値は JSON 配列文字列で `["Bizreach"]` / `["Linkedin"]` / `["dodaX"]` 等（※選択肢名は `Linkedin`。大文字小文字に注意）
- `Ver`（number）：版番号
- `使用中`（checkbox）：使うなら `"__YES__"`、止めるなら `"__NO__"`（チェックボックスは `__YES__`/`__NO__` 文字列で渡す）
- `再送用`（checkbox）：新規検証は `"__NO__"`、非返信者への再送ログは `"__YES__"`（§返信率の集計を汚さないため分離）
- `ポジション`（relation）：ポジションページの URL を**そのまま**渡す（例：`https://app.notion.com/p/...`）。**二重引用符で包まない**（包むと `Invalid agent URL` で全体が落ちる）
- `送信数`/`返信数`（number）：新規は `0`
- `備考`（text）：版の意図・斉藤FB の反映点・出典・報酬補足・送信者（古市/佐藤）を残す

これで `クライアント`・`注力`・`返信率`・`テンプレート名` 等のロールアップ/数式は自動算出される（手で触らない）。

### 9-2. ポジション直下に「まとめページ」を作成

ポジションページ（ポジションDB の該当ページ）の**子ページ**として作る。**このページは Claude.ai プロジェクトに md として取り込み、対象者ごとにスカウト文をパーソナライズする「源泉」**になる。したがって、どこを個別最適するか（個別化スロット）と、経歴が読めない時の汎用退避版を**必ず残す**（汎用版だけにしない）。`notion-create-pages` で parent＝`page_id: {ポジションページID}`、プロパティは `title` のみ。

- タイトル：`［スカウト文］{ポジション名}｜{社名}｜{YYYY-MM-DD}-vN`
- 構造：**媒体ごとにトグル見出し**＋その中に `## タイトル`／`## 本文（個別化スロット入り）`／`## 汎用退避版`（content の書式）：

```
---
# ビズリーチ {toggle="true"}
	## タイトル
	{件名}
	## 本文（個別化スロット入り）
	{本文。冒頭に個別化スロット 例:{現職領域}/{具体実績＝…}、媒体内送信なので <br> で改行}
	## 汎用退避版（経歴が読めない一斉送信時・スロットは推測で埋めない）
	{冒頭の連絡理由のみ差し替えた汎用版}
# Linkedin {toggle="true"}
	## タイトル
	...
# dodaX {toggle="true"}
	## タイトル
	...
```

- トグル直下は**タブ字下げ**で入れ子にする（Notion enhanced markdown。fetch で round-trip 確認済みの書式）。
- 宛名プレースホルダは媒体慣習に合わせる：Bizreach＝`{NAME}様`、LinkedIn＝`{姓}様`、dodaX＝`[候補者名]様`。
- 各媒体トグルに **`## 本文（個別化スロット入り）`＋`## 汎用退避版` をセットで載せる**。スロット（`{現職領域}`／`{具体実績＝…}`）＝Claude.ai が対象者ごとに最適化する場所、退避版＝経歴が読めない時の安全版。**DBレコード側も同じ構造（スロット＋退避版）を保持する。**
- プレースホルダの波括弧・角括弧は**エスケープして渡す**（`\{NAME\}`、`\{姓\}`、`\[候補者名\]`）。

### 9-3. 改訂（v を重ねる）時の作法

同じポジションで改訂したら、**旧版レコードは消さず残す**（送信実績＝返信率ベースラインの保全）。新版は別レコード＋別まとめページ（日付・vN 違い）として追加し、`使用中` の付け替えで「いま使う版」を表す。旧版は誤送信防止のため `使用中＝__NO__` にしておくと安全。

## 10. スカウトPDCA（送信量・検証・打ち手の標準ループ）

新ポジションでスカウトを回す時の標準サイクル。「何通送って・どう判定して・いつ打ち手を変えるか」を固定する。本セッション（ギフティ事業開発）の運用を汎用化したもの。

### 10-1. 1サイクルの流れ

Plan（ターゲット＋文面）→ Do（送信）→ Check（返信率を読む）→ Act（増量／改修／再送／ピボット）→ 次サイクル。

### 10-2. Do：送信の単位と量

- **1セグメント（ポジション×ペルソナ×文面版）に集中**し、まず**各40〜50通**積む（返信率を読める最小サンプル）。
- 目安は**20〜30通/日 ×1セグメント**。個別化スロットは Claude が下書き量産して、質を落とさず量を出す。
- **複数セグメントに薄く分散しない**（どれも読める単位に届かず、いつまでも判定できない）。

### 10-3. Check：返信率で判定

- 各セグメント40〜50通到達で初期判定。**返信は遅れて来る**（execは即返信しない・週末は動かない）ため、送信完了から**数営業日〜1週間**待って確定する。
- 閾値（目安）：
  - **返信率 8%超 → 当たり。量を維持／増量して回す。**
  - **3%未満 → 止めて打ち手変更（文面 or ターゲット）。量を増やさない**（弱い文面で狭い母集団を焼かない＝企業法務リーガル0/120の教訓）。
  - **3〜8%（グレー）→ 個別化強化・角度変えで再検証。**

### 10-4. Act：打ち手変更

- **文面が原因の仮説** → v2（改訂版）を作り、**新規候補で**測る。旧版レコードは消さずベースラインとして残す（§9-3）。
- **ターゲットが原因の仮説** → 母集団・要件・ペルソナ（P1/P2/P3）を見直す。
- **検証は必ず新規候補で**。同じ人に v1→v2 を重ねると「文面が効いた」のか「2回目だから」かが分離できない。

### 10-5. 再送（非返信者への再アプローチ）

- スカウトDBの **`再送用=YES` で別集計**（本体の返信率を汚さない）。
- **2〜3週間空ける／同一人物2〜3回まで／磨くのは言い回しより"角度"**（別ポジション提案・新しい具体ネタ）。**高フィットの非返信者に限定**（媒体の枠・クレジットを浪費しない）。

### 10-6. 量の見積り（ファネル全体）

- **送信→決定 ≒ 1,400〜1,500通／1決定**（佐藤の歴史的レート。返信率を倍化できれば 700〜1,200 に短縮）。出発点は `返信率5.17% × 返信→入社1.3%`。レイヤー・クライアントで変動するため目安として使う。
- **リードタイム**：新規スカウトの果実は数ヶ月先。年内決定を狙うなら送信を前倒し、それ以外は次期パイプラインの仕込みと割り切る。
- **返信率は入口の最大レバーだが、決定率は下流も効く**：アプライ→書類通過（マッチング精度＝`client-profile.md §7` の5軸）と後段クロージング（買い込み・辞退の先回り）。返信率だけを追わない。

### 10-7. 記録先

- **送信/返信**：Notion スカウトDB（媒体ごとレコード／`送信数`・`返信数`・`返信率`）＝§9-1。
- **文面**：ポジション直下のまとめページ＝§9-2。
- **週次の進捗**：セグメント別返信率・各案件のEVを1枚のトラッキング表（Craft）に残し、毎週更新する。

### 10-8. 判断フロー（要約）

1セグメント選ぶ → 40〜50通（20〜30/日）→ 返信率を読む → **8%↑なら増量／3%↓なら打ち手変更（v2 or ターゲット見直し）／中間は個別化・角度で再検証** → 勝ち版が出たら横展開＋非返信者へ角度を変えて再送。
