# スカウト評価キット生成エージェント

Claude.ai プロジェクトナレッジ用のスカウト評価ファイル一式を生成するためのエージェント。
BizReach 等で発見した**未接触候補**のスカウト適性評価を Claude.ai 側で運用する前提で、ポジション単位の評価資料パッケージを Craft / Notion / SF / Google Drive から横断生成する。

---

## 1. 用途と前提

- **対象シーン**：未接触候補（Craft/SF に履歴なし）のスカウト適性を Claude.ai プロジェクトでコスト効率よく評価したいケース
- **既接点候補**（Craft/SF に candidate md あり）の評価はこのキットを使わず、Claude Code から直接 profile md を読み込んで評価する
- 評価ログの蓄積・月次リセット運用は `agents/scout-kit.md` の「6. Claude.ai 側の運用」参照

## 2. 起動方法

ユーザーが以下のいずれかの形で依頼した時に起動：

- 「scout-kit を使って {社名}／{ポジション} で生成」
- 「{社名} の {ポジション} のスカウト評価キットを作って」
- 「Claude.ai 用の評価ファイル4点を {社名}／{ポジション} で揃えて」
- **「scout-kit {ポジション}-{社コード}」**（短縮形、社コードによる指定）
- **「{ポジション}-{社コード} のキット作って」**

差分更新（既存ファイルあり）の場合は末尾に「diff 形式で更新」と付ける運用。

### 2.1 社コード → 社名 の解決

ユーザーが社コードで指定してきた場合（例：`CISO候補-1234`）、着手前に以下のいずれかで社名を解決する：

1. **Google Drive** `01_企業情報/`（folder ID: `1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx`）配下の
   `{社名} #{社コード}/` フォルダ名から逆引き（`mcp__gdrive__search_files` で `#{社コード}` を含むフォルダを検索）
2. **Notion 企業 DB** で `社コード` プロパティ一致のページを `mcp__notion-search` で検索し、`社名` プロパティを取得
3. **SF** の Account オブジェクトで `salesforce_query_records` により社コード一致を検索

解決できなかった場合はユーザーに「該当する社名が特定できませんでした。社名で指定し直してください」と確認する。**推測で進めない**。

## 3. 参照ソース（必ず全て fetch してから着手）

CLAUDE.md セクション8 のクライアント／ポジションプロファイル自動読み込みルールを踏襲：

1. **Craft `12_Client｜企業/{社名}/{社名}.md`**（クライアント本体）
2. **Craft `12_Client｜企業/{社名}/positions/{ポジション}.md`**（ポジション md）
3. **Notion 企業 DB・ポジション DB・スカウト DB** で `{社名}` / `{ポジション}` 関連ページを検索
4. **Notion 1on1 DB**（`collection://22c7d017-b6a0-8033-837e-000bd6c2ae8b`）直近6件から `{社名}` / `{ポジション}` の言及を抽出
5. **Google Drive `01_企業情報/{社名} #{社コード}/`**（folder ID: `1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx` 配下）の最新ファイル
6. **SF** で過去推薦履歴（推薦済み・通過/見送り結果込み）を `salesforce_query_records` で取得

## 4. 出力先

**Craft が正本**。`work > 10_Recruitment > 12_Client｜企業 > {社名}/positions/` 配下に新規フォルダ **`{ポジション}-scout-kit/`** を切り、その中に 6 ドキュメント（ターゲット人物像・要件・経営陣スタンス・過去推薦パターン・評価フレーム・評価履歴）を作成する。

- フォルダ名は社名指定でも社コード指定でも `{ポジション}-scout-kit` で統一（ポジション md と同階層に並ぶ）
- 各ドキュメントの冒頭 callout に**社名・社コード・ポジション名・最終更新日・ソース**を併記して検索性を確保
- git リポジトリ側には**置かない**。一時環境で消える上、Craft とで二重管理になり同期事故の原因になる

### 4.0 Craft 書き込み時の注意（`agents/craft-writing.md` を必ず読む）

- bullet を含むセクションは**実改行 `\n\n` をブロック区切りとして送ると bullet スタイルが保持される**（要件・経営陣スタンス・過去推薦パターン等で使用）
- テーブルは1ブロックで送って OK（実改行で行区切り）
- 大きな1ショット送信ではなく、見出し＋関連 bullet 群の**セクション単位**で送ると失敗時の復旧が早い
- 書き込み後は `blocks get` で各ドキュメントの構造を確認する

### 4.0 `ターゲット人物像.md` ★ペルソナの起点

スカウトの「誰を狙うか」を定義し、`agents/scout.md` のペルソナ別作成（P1/P2/P3）と訴求設計の入力になる。Craft の標準プロンプト **「ターゲット人物像生成用プロンプト（リクナビ）」**（block ID: `BA6AF6F2-FA37-4E73-9F8A-6E0696BDC04F`）の出力形式に準拠する。

- **リソースは事前格納ではなく §3 の参照ソースから横断取得する**（プロンプト本文は ChatGPT プロジェクトへの事前格納を前提に書かれているが、ここでは Craft/Notion/SF/GDrive/1on1 から取得した内容を文脈として与える）
- 出力構成：`Tier1（最優先）／Tier2（次点）／必要時 Tier3`。各 Tier に **想定年齢層・現在の立ち位置・主な経験・今回の求人とフィットする理由・本人が感じていそうな課題**を記述
- 末尾に **「スカウト・面談設計上の示唆」**：`Tier1 に刺さる訴求軸／Tier2 に説明すべき不安点／面談で深掘る論点`
- **企業特定の回避**：会社名・サービス名・固有プロダクト名・数値は出さない（外部に出ても特定されない粒度）
- Tier は「優秀さ」ではなく「今回の求人との適合度・即効性」で分ける
- **scout.md ペルソナへの接続**：各 Tier の想定年齢層・訴求軸を P1（売れ線・上り）/P2（中堅マネジメント）/P3（シニア・エグゼク）にマッピングし、スカウト作成時の構成・件名・CTA を決める

### 4.1 `要件.md`

- ポジション概要（1段落）
- Must 要件（箇条書き、優先度明示）
- Want 要件（箇条書き）
- 年収レンジ（提示可能上限・想定中央値）
- 事業フェーズ／組織状況
- 選考フロー（回数・面接官・想定リードタイム）
- レポートライン／チーム構成

### 4.2 `経営陣スタンス.md`

- 意思決定者の氏名・役職・人物像（公開可能範囲で）
- 重視するポイント（カルチャー・実績・スタイル）
- NG パターン（過去にダメ出しされた候補者タイプ）
- コミュニケーション上の留意点（レス速度・温度感）
- 1on1 議事録で言及された**最新の方針変化**があれば反映

### 4.3 `過去推薦パターン.md` ★最重要

- 過去推薦者の通過/見送り結果を**イニシャル化**して列挙
- 「通った人の共通点」3〜5項目
- 「落ちた人の共通点」3〜5項目
- フィードバックから読み取れる**暗黙の選好**（要件書に書かれていない判断軸）
- ※実名・所属が特定されないよう、現職は**業界カテゴリ粒度**に丸める

### 4.4 `評価フレーム.md`

- `agents/client-profile.md` セクション7「マッチング評価」の5軸（要件適合・カルチャー適合・年収適合・フェーズ適合・過去パターン適合）をこのポジション専用にカスタマイズ
- 各軸の採点基準（5段階、各点の定義）
- 出力フォーマット指定：
  - 評価サマリ（200字以内）
  - 5軸スコア表
  - 強調点・懸念点・推奨アクション
  - **末尾に「評価履歴.md 追記用1行ブロック」**（評価時点で分かる列のみ出力）：
    `評価日｜媒体ID｜現職カテゴリ｜総合点｜判定(A/B/C/D)｜一言所感｜年代(P#)`
  - **媒体ID＝媒体ごとの安定ID**を必ず使う。フリーな頭文字（"S.M." 等）は SF と照合できず匿名性も中途半端なので**不可**。これが SF Contact と突合する join キー。媒体別の値は以下：
    - **BizReach** → 会員ID（例：`BU5854836`）
    - **LinkedIn** → プロフィールURL の vanity slug（例：`linkedin.com/in/yuta-sato-xxxx` の `yuta-sato-xxxx`）。LinkedIn には BizReach のような匿名会員IDが無いため、slug を会員IDの代替とする。一意で SF と照合でき（SF Contact に LinkedIn URL を保存しておく）、評価ログに実名そのものを載せずに済む。**slug が取得できない場合のみ氏名で代替**する（同姓同名衝突・実名露出のリスクがあるためフォールバック扱い）
    - **その他媒体**（dodaX 等） → 各媒体の安定ID。無ければ氏名（フォールバック）
  - **年代(P#)** は年代バンド＋想定ペルソナ（例：`30代前半/P1`）。層別分析の主軸（P1/P2/P3 は `agents/scout.md` セクション2）。既存ログとの列ズレを避けるため**末尾に追加**し、旧来の6列（評価日・媒体ID・現職カテゴリ・総合点・判定・一言所感）の並びは保つ
  - 評価履歴.md の正本テーブルは上記7列にさらに後続列 `送信日｜返信日｜結果｜SF id` を加えた全11列。埋めるタイミングは3段階：**①評価時**＝プロンプトが上記7列を出力／**②送信時**＝佐藤が `送信日` を追記（"送ったか"＝返信率の分母）／**③返信時**＝Claude Code が `返信日・結果・SF id` を書き戻す（§6.5）
  - `判定`（A/B/C/D）は**送信推奨度**であって "送信した" ではない。実送信は `送信日` の有無で表す（判定列・所感列に送信判断を混ぜない）
  - **ポジション固有の追加列は許容する**：上記11列を「最低限の共通列」とし、ポジション固有のシグナル列（例：企業法務の `枠判定`＝リーダー枠/メンバー枠）が必要なら、標準列の後・アウトカム列（送信日以降）の前に挿入してよい。層別分析は列の位置ではなく**列名で結合**する前提（例：企業法務は枠判定を含む12列）

## 5. 取り扱い注意

- 進行中候補者の実名・現職名は伏せる（イニシャル＋業界カテゴリで記述）
- 機密度高の情報（年収上限・経営陣の NG 発言等）はファイル冒頭に
  `※Claude.ai 外への共有禁止` と明記
- 情報不足の項目は `[要確認：◯◯]` と残し、**推測で埋めない**
- 完了後、生成した5ドキュメントの Craft リンクを一覧表示し、Claude.ai プロジェクトへのアップロード手順を1行で添える

## 6. Claude.ai 側の運用

### 6.1 プロジェクト構成方針（会社単位を推奨）

**基本：会社単位の統合プロジェクト**を作り、同一企業の複数ポジションのスカウトキットをまとめて格納する。理由は以下：

- ヘッドハンティング業務では「同一企業の別ポジションへの横展開」が頻繁（CISO 不適合 → CIO 候補に振り替え等）
- 議事録・経営陣スタンス・企業概要などの**会社共通情報を1箇所で管理**できる
- 議事録更新がN箇所更新にならず運用負荷が低い

**構成判断基準：**

| 構成 | 向いているケース | 注意点 |
|---|---|---|
| 会社単位（推奨デフォルト） | 同一企業に複数ポジション、議事録更新が定期的、横展開判断したい | ファイル命名でポジション識別が必須 |
| ポジション単位（純化型） | 会社にポジション1つだけ／選考プロセスがポジション間で完全に別系統／1ポジションあたり評価候補 50件超 | 会社情報の重複管理コスト |

**会社単位プロジェクトのファイル構成例：**

```
{社名}スカウト評価プロジェクト/
├── 企業概要.md                       ← 会社共通
├── 経営陣スタンス.md                 ← 会社共通
├── 議事録_2025-Q4.md                 ← 会社共通（※社外共有禁止）
├── [CISO候補] 要件.md
├── [CISO候補] 過去推薦パターン.md
├── [CISO候補] 評価フレーム.md
├── [CISO候補] 評価履歴.md
├── [CIO候補] 要件.md
├── [CIO候補] 過去推薦パターン.md
├── [CIO候補] 評価フレーム.md
└── [CIO候補] 評価履歴.md
```

- ポジション固有ファイル（Craft `{ポジション}-scout-kit/` 由来）には **`[ポジション名]` プレフィックス**を必ず付けてアップロードする
- 会社共通ファイル（経営陣スタンス・議事録・企業概要）はプレフィックスなし
- 機密度の高い議事録は冒頭に `※社外共有禁止` を明記

**会社単位用カスタム指示テンプレ：**

```
候補者評価リクエストを受けたら、最初に対象ポジション名を必ず確認すること。
評価では `[ポジション名]` プレフィックスのファイル群＋プレフィックスなしの会社共通ファイルのみを参照し、
他ポジションのファイルは混在参照しないこと。

評価は [ポジション名] 評価フレーム.md の5軸で採点し、
末尾に「評価履歴.md 追記用ブロック」を出力する。
```

### 6.2 初期セットアップ（1回）

**会社単位プロジェクト（推奨）の場合：**

1. Claude.ai でプロジェクト作成（例：「{社名}スカウト評価」）。既存プロジェクトがあれば新ポジション分を追加するだけで OK
2. Craft の `{ポジション}-scout-kit/` 配下の5ドキュメントを md エクスポート（Craft アプリの「Share → Export as Markdown」）
3. アップロード時にファイル名へ **`[ポジション名]`** プレフィックスを付与
4. 会社共通ファイル（`経営陣スタンス.md` / `企業概要.md` / 議事録）は初回のみアップロード（既存の場合は更新差分のみ差し替え）
5. プロジェクトのカスタム指示に 6.1 の「会社単位用カスタム指示テンプレ」を記載

**ポジション単位プロジェクトの場合：**

1. Claude.ai でプロジェクト作成（例：「{社名}_{ポジション}_スカウト評価」）
2. Craft の `{ポジション}-scout-kit/` 配下の5ドキュメントをそのままアップロード（プレフィックス不要）
3. プロジェクトのカスタム指示に以下を記載：
   ```
   候補者情報が貼られたら、評価フレーム.md の5軸で採点し、
   末尾に「評価履歴.md 追記用ブロック」を出力すること。
   ```

### 6.3 日々の評価

1. 新規チャットで候補者プロフィール（BizReach 等）を貼る
2. **会社単位プロジェクトの場合は対象ポジション名を冒頭で明示**（例：「[CISO候補] の評価をお願い」）
3. Claude が評価＋追記ブロックを出力
4. 追記ブロックをコピー →プロジェクトナレッジの該当 `評価履歴.md`（会社単位なら `[ポジション名] 評価履歴.md`）を編集モードで開き末尾にペースト → 保存

### 6.4 月次リセット

1. ポジション単位で `評価履歴.md`（会社単位なら `[ポジション名] 評価履歴.md`）の中身を全選択コピー
2. Claude Code に「`評価履歴.md` の中身を Craft `{社名}/positions/{ポジション}-scout-kit/scout_log_{YYYY-MM}.md` に書き込んで」と指示
3. プロジェクトの該当 `評価履歴.md` を空テンプレに戻して再保存
4. 会社単位プロジェクトでは、ポジション毎にこの手順を繰り返す

### 6.5 移行トリガー（＋アウトカム書き戻し）

- 候補者から返信が来た時点で、Claude Code 側で candidate md を新規作成し以降は正本連動運用へ移行する
- Claude.ai プロジェクトは**未接触プール専用**と割り切る

**アウトカム書き戻し（評価履歴を学習資産にする要）：** 返信→SF登録→Notion 自動連携→Claude Code でプロファイル作成、という導線に相乗りして、評価ログへ結果を書き戻す。専用の追記作業を作らず、プロファイル作成時の1ステップとして実行する（手順は `agents/candidate-profile.md` §5.2 に組み込み済み）。

- **照合キー：** 評価履歴の `媒体ID`（BizReach＝会員ID／LinkedIn＝プロフィールURL の slug／その他＝媒体の安定ID。§4.2 参照）。**媒体ID は SF Contact に保存されている**（BizReach 会員ID・LinkedIn URL 等。Notion 候補者には現状なし）。プロファイル作成時に SF を取得するので、SF の媒体IDで評価履歴の `媒体ID` 列と照合する。LinkedIn 由来は SF の LinkedIn URL の slug で突合する。SF にも該当IDが無い場合のみ氏名で曖昧照合（コスト・誤りが増えるので非推奨）。Notion 側にも媒体IDを持たせれば冗長性は増すが必須ではない
- **書き戻し先：** Claude.ai 側の live `評価履歴.md` は Claude Code から書けないため、**Craft の `scout_log_{YYYY-MM}.md`（月次アーカイブ＝正本台帳）**を更新する。照合は媒体ID＋直近2〜3ヶ月分の scout_log に絞る
- **更新内容：** 該当行の後続列に `返信日｜結果（面談/推薦/通過/見送り/辞退）｜SF id` を追記（返信そのものは返信日の有無で表す）
- **コスト：** プロファイル作成で既に SF を取得しているため追加トークンは小さい。肝はトークンではなく媒体IDによる行の特定

## 7. 差分更新モード

既存の Craft `{社名}/positions/{ポジション}-scout-kit/` がある状態で「diff 形式で更新」と指示された場合：

1. 既存5ドキュメントを `blocks get` で取得
2. Craft / Notion / 1on1 / SF を再 fetch し、新しい情報・変化点を抽出
3. 各ドキュメントに対し、変更箇所のみを **before/after の diff ブロック** で提示
4. ユーザー承認後に `blocks update` / `blocks add` で反映
5. 反映後、Claude.ai プロジェクト側で「該当ドキュメントを Craft から再エクスポートして差し替えが必要」と1行アナウンス
