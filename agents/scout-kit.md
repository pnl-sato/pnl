# スカウト評価キット生成エージェント

Claude.ai プロジェクトナレッジ用のスカウト評価ファイル一式を生成するためのエージェント。
BizReach 等で発見した**未接触候補**のスカウト適性評価を Claude.ai 側で運用する前提で、ポジション単位の評価資料パッケージを Craft / Notion / SF / Google Drive から横断生成する。

---

## 1. 用途と前提

- **対象シーン**：未接触候補（Craft/SF に履歴なし）のスカウト適性を Claude.ai プロジェクトでコスト効率よく評価したいケース
- **既接点候補**（Craft/SF に candidate md あり）の評価はこのキットを使わず、Claude Code から直接 profile md を読み込んで評価する
- 評価ログの蓄積・書き戻し運用は Notion DB「スカウト評価ログ」（§4.5）と「6. Claude.ai 側の運用」参照

## 2. 起動方法

ユーザーが以下のいずれかの形で依頼した時に起動：

- 「scout-kit を使って {社名}／{ポジション} で生成」
- 「{社名} の {ポジション} のスカウト評価キットを作って」
- 「Claude.ai 用の評価ファイル4点を {社名}／{ポジション} で揃えて」
- **「scout-kit {ポジション}-{社コード}」**（短縮形、社コードによる指定）
- **「{ポジション}-{社コード} のキット作って」**

差分更新（既存ファイルあり）の場合は末尾に「diff 形式で更新」と付ける運用。

### 2.1 社コード → 社名 の解決

ユーザーが社コードで指定してきた場合（例：`CISO候補-1234`）、着手前に以下のいずれかで社名を解決する：

1. **Google Drive** `01_企業情報/`（folder ID: `1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx`）配下の
   `{社名} #{社コード}/` フォルダ名から逆引き（`mcp__gdrive__search_files` で `#{社コード}` を含むフォルダを検索）
2. **Notion 企業 DB** で `社コード` プロパティ一致のページを `mcp__notion-search` で検索し、`社名` プロパティを取得
3. **SF** の Account オブジェクトで `salesforce_query_records` により社コード一致を検索

解決できなかった場合はユーザーに「該当する社名が特定できませんでした。社名で指定し直してください」と確認する。**推測で進めない**。

## 3. 参照ソース（必ず全て fetch してから着手）

CLAUDE.md セクション8 のクライアント／ポジションプロファイル自動読み込みルールを踏襲：

1. **Craft `12_Client｜企業/{社名}/{社名}.md`**（クライアント本体）
2. **Craft `12_Client｜企業/{社名}/positions/{ポジション}.md`**（ポジション md）
3. **Notion 企業 DB・ポジション DB・スカウト DB** で `{社名}` / `{ポジション}` 関連ページを検索
4. **Notion 1on1 DB**（`collection://22c7d017-b6a0-8033-837e-000bd6c2ae8b`）直近6件から `{社名}` / `{ポジション}` の言及を抽出
5. **Google Drive `01_企業情報/{社名} #{社コード}/`**（folder ID: `1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx` 配下）の最新ファイル
6. **SF** で過去推薦履歴（推薦済み・通過/見送り結果込み）を `salesforce_query_records` で取得

## 4. 出力先

**Craft が正本**。`work > 10_Recruitment > 12_Client｜企業 > {社名}/positions/` 配下に新規フォルダ **`{ポジション}-scout-kit/`** を切り、その中に 5 ドキュメント（ターゲット人物像・要件・経営陣スタンス・過去推薦パターン・評価フレーム）を作成する。**評価履歴は Craft md ではなく Notion DB「スカウト評価ログ」に移行した（2026-06-12。仕様は §4.5）**——評価のたびに Claude.ai が Notion に1行 create し、返信後に Claude Code が同行を更新する一層運用。

- フォルダ名は社名指定でも社コード指定でも `{ポジション}-scout-kit` で統一（ポジション md と同階層に並ぶ）
- **各ドキュメントのタイトルは `[{ポジション名}-{社コード}] {種別}` 形式で作成する**（種別＝ターゲット人物像／要件／経営陣スタンス／過去推薦パターン／評価フレーム。例：`[Head of HR・People Director-IGS] 要件`）。Claude.ai は md をフォルダ階層なしのフラットなファイル名で扱うため、別社・別ポジションのファイルと混在しても識別できるよう、プレフィックスを**ファイル名自体に焼き込む**（§6 のアップロード時に手で付け直さない）。社コードは §2.1 で確定したもの（Drive `01_企業情報/{社名} #{社コード}/` のコード）を使い、暫定なら確定後にリネームする。Craft 正本をこの名前で作っておけば、md エクスポートがそのまま正しいファイル名になる。
  - 新規作成：`documents create --title "[{ポジション名}-{社コード}] {種別}" --folder <scout-kitフォルダID>`（角括弧 `[ ]` はそのまま literal で入る）。
  - 既存をリネーム（プレフィックス後付け）：`blocks update --id <docのrootBlockId> --markdown "[{ポジション名}-{社コード}] {種別}"`（タイトルだけ置換され本文・callout・表は保持される。2026-06-17 検証済み）。
  - **Google Drive 側はコネクタにリネーム/移動/削除ツールが無い**（`copy_file`/`create_file` のみ）。Drive 上のファイルを直接改名できないため、改名は必ず Craft 正本で行い、md を再エクスポートして Drive/Claude.ai 側を差し替える（Drive で手リネームしようとしない）。
- 各ドキュメントの冒頭 callout に**社名・社コード・ポジション名・最終更新日・ソース**を併記して検索性を確保
- git リポジトリ側には**置かない**。一時環境で消える上、Craft とで二重管理になり同期事故の原因になる

### 4.0 Craft 書き込み時の注意（`agents/craft-writing.md` を必ず読む）

- bullet を含むセクションは**実改行 `\n\n` をブロック区切りとして送ると bullet スタイルが保持される**（要件・経営陣スタンス・過去推薦パターン等で使用）
- テーブルは1ブロックで送って OK（実改行で行区切り）
- 大きな1ショット送信ではなく、見出し＋関連 bullet 群の**セクション単位**で送ると失敗時の復旧が早い
- 書き込み後は `blocks get` で各ドキュメントの構造を確認する

### 4.0 `ターゲット人物像.md` ★ペルソナの起点

スカウトの「誰を狙うか」を定義し、`agents/scout.md` のペルソナ別作成（P1/P2/P3）と訴求設計の入力になる。Craft の標準プロンプト **「ターゲット人物像生成用プロンプト（リクナビ）」**（block ID: `BA6AF6F2-FA37-4E73-9F8A-6E0696BDC04F`）の出力形式に準拠する。

- **リソースは事前格納ではなく §3 の参照ソースから横断取得する**（プロンプト本文は ChatGPT プロジェクトへの事前格納を前提に書かれているが、ここでは Craft/Notion/SF/GDrive/1on1 から取得した内容を文脈として与える）
- 出力構成：`Tier1（最優先）／Tier2（次点）／必要時 Tier3`。各 Tier に **想定年齢層・現在の立ち位置・主な経験・今回の求人とフィットする理由・本人が感じていそうな課題**を記述
- 末尾に **「スカウト・面談設計上の示唆」**：`Tier1 に刺さる訴求軸／Tier2 に説明すべき不安点／面談で深掘る論点`
- **企業特定の回避**：会社名・サービス名・固有プロダクト名・数値は出さない（外部に出ても特定されない粒度）
- Tier は「優秀さ」ではなく「今回の求人との適合度・即効性」で分ける
- **scout.md ペルソナへの接続**：各 Tier の想定年齢層・訴求軸を P1（売れ線・上り）/P2（中堅マネジメント）/P3（シニア・エグゼク）にマッピングし、スカウト作成時の構成・件名・CTA を決める

### 4.1 `要件.md`

- ポジション概要（1段落）
- Must 要件（箇条書き、優先度明示）
- Want 要件（箇条書き）
- 年収レンジ（提示可能上限・想定中央値）
- 事業フェーズ／組織状況
- 選考フロー（回数・面接官・想定リードタイム）
- レポートライン／チーム構成

### 4.2 `経営陣スタンス.md`

- 意思決定者の氏名・役職・人物像（公開可能範囲で）
- 重視するポイント（カルチャー・実績・スタイル）
- NG パターン（過去にダメ出しされた候補者タイプ）
- コミュニケーション上の留意点（レス速度・温度感）
- 1on1 議事録で言及された**最新の方針変化**があれば反映

### 4.3 `過去推薦パターン.md` ★最重要

- 過去推薦者の通過/見送り結果を**イニシャル化**して列挙
- 「通った人の共通点」3〜5項目
- 「落ちた人の共通点」3〜5項目
- フィードバックから読み取れる**暗黙の選好**（要件書に書かれていない判断軸）
- ※実名・所属が特定されないよう、現職は**業界カテゴリ粒度**に丸める

### 4.4 `評価フレーム.md`

- `agents/client-profile.md` セクション7「マッチング評価」の5軸（要件適合・カルチャー適合・年収適合・フェーズ適合・過去パターン適合）をこのポジション専用にカスタマイズ
- 各軸の採点基準（5段階、各点の定義）

#### 経歴推定レイヤー（スパース profile 補完）★LinkedIn 対策

**LinkedIn 等で職務内容の記述が無く、企業名・役職・在籍年月しか読めない候補者**を採点する前に、まず「この役職・この年数なら標準的にこのくらいやっているはず」を構造的に推定し、要件との突き合わせに使う。5軸採点の**前段**に置く（推定したスコープを各軸の判断材料にする）。

推定の手順（3軸 → 要件突き合わせ）：

1. **役職レベルの読み替え** — 役職名を責任レイヤー（担当者／リーダー／マネージャー／部長／執行役員／CxO）に正規化し、そのレイヤーが標準的に持つ責任範囲（管掌人数・予算規模・意思決定権・対外折衝の有無）を当てる。同じ「マネージャー」でも大企業とスタートアップで実態が違うので、§3 で取得した企業フェーズ・規模で必ず補正する。
2. **在籍年数の読み替え** — 年数を「その役割で成果が出るだけの時間があったか」に変換する。目安は〜1年＝立ち上げ／見極め途中（成果未確定）、2〜3年＝一定の完遂経験あり、4年以上＝深い専門性か同一役割の固定化（伸びと停滞の両面）。役職の刻み（年数に対し役職が上がっているか）も昇進スピードのシグナルとして読む。
3. **企業フェーズ・規模での割り戻し** — 上の2軸を企業フェーズ（シード／成長／上場／大企業）と規模で補正する。例：スタートアップの部長＝少人数で広く一気通貫、大企業の部長＝縦割りで深いが範囲は限定、という具合にスコープの「広さ／深さ」を読み分ける。
4. **要件・ターゲット人物像との突き合わせ** — 推定したスコープを `要件.md` の Must/Want と `ターゲット人物像.md`（§4.0）の Tier 別「主な経験」に重ね、「この役職・年数なら Must の◯◯は経験済みと推定／△△は未経験の可能性」と各軸の採点に接続する。

確信度と扱い（CLAUDE.md §6 の出典分離・ハルシネーション原則を厳守）：

- 推定にはすべて **確信度（高／中／低）** を付け、**`【推定】` タグを明示**する。profile に明記のある一次情報（【確認済み】）と、役職・年数から補完した推定（【推定】）を分けて出す。
- **推定を事実として断定しない／スカウト本文に推定を事実として書かない。** 推定は評価の解像度を上げるためのもので、スカウト文に載せる固有実績は profile 明記のものに限る（推定をパーソナライズ実績に転用しない）。
- **スコアへの反映は確信度で減衰させる**：推定ベースの加点は確信度に応じて割り引き、**確信度「低」はスコアに乗せず面談検証論点へ回す**。逆に情報不足を理由に減点もしない（読めない＝不利、にせず「要面談確認」のグレー扱いにする）。
- 確信度「中〜低」の推定は **「面談で確認すべき論点」** として出力に必ず残し、一次接触後の見極め（裏取り質問）へ引き継ぐ。

- 出力フォーマット指定：
  - 評価サマリ（200字以内。**冒頭で【確認済み】と【推定】を1行ずつ分けて要約**する）
  - **推定スコープ欄**（役職×年数×フェーズから想定した責任範囲）＋各推定の確信度（高／中／低）
  - 5軸スコア表（推定に依存した軸は確信度で減衰させた点数とし、欄外に `【推定】` 注記）
  - 強調点・懸念点・推奨アクション
  - **面談で確認すべき論点**（確信度 中〜低の推定を裏取りする質問。スパース profile ほどここが厚くなる）
  - **評価ログDBへの登録**（正本＝Notion。仕様は §4.5）：評価が固まったら、**Claude.ai が `スカウト評価ログ` DB に1行 create** する（`notion-create-pages`、data source `437a2bad-94d6-4397-8dc4-1aa986a34c3d`）。Notion 書込が承認待ちで止まった時（CLAUDE.md §14）のフォールバックとして、末尾に §4.5 の「登録ブロック（パイプ1行）」もテキストで必ず残す
  - **媒体ID＝媒体ごとの安定ID**を必ず使う（DB の title）。フリーな頭文字（"S.M." 等）は SF と照合できず匿名性も中途半端なので**不可**。SF Contact と突合する join キー。媒体別の値は以下：
    - **BizReach** → 会員ID（例：`BU5854836`）
    - **LinkedIn** → プロフィールURL の vanity slug（例：`linkedin.com/in/yuta-sato-xxxx` の `yuta-sato-xxxx`）。LinkedIn には BizReach のような匿名会員IDが無いため slug を代替とする。一意で SF と照合でき、評価ログに実名そのものを載せずに済む。**slug が取得できない場合のみ氏名で代替**（同姓同名衝突・実名露出のリスクがあるためフォールバック扱い）
    - **その他媒体**（dodaX 等） → 各媒体の安定ID。無ければ氏名（フォールバック）
  - **ペルソナ(P#)・年代バンド** は層別分析の主軸（P1/P2/P3 は `agents/scout.md` セクション2）。DB では `ペルソナ`／`年代バンド` の別カラムに分離済（旧 md の `年代(P#)` 結合表記＝`30代前半/P1` は廃止）
  - **推定heavy**（checkbox）：§4.4 の経歴推定レイヤーを多用した（職務記述が薄い）評価で `__YES__`。後で「推定heavy 評価の返信率／精度」を層別検証するためのキー
  - `判定`（A/B/C/D）は**送信推奨度**であって "送信した" ではない。実送信は `送信日` の有無で表す（判定・所感に送信判断を混ぜない）
  - **ポジション固有シグナル**（例：企業法務の `枠判定`＝リーダー枠/メンバー枠）は DB の `追加シグナル`（text）に入れる。md 時代のような列追加は不要（単一DBに集約され、層別は `追加シグナル` の値で絞る）

### 4.5 評価ログ（Notion DB＝正本台帳）★Craft md から移行（2026-06-12）

- **DB**：`スカウト評価ログ`（親ページ Recruitment）
  - database：`42e00d7d-5c7e-4b09-aa80-d6c1cd4e55bf`
  - data source（create-pages / update-page の parent・照合対象）：`437a2bad-94d6-4397-8dc4-1aa986a34c3d`

| SF id | text | 文字列 | ③で追記 |
| スカウト文／ポジション／候補者 | relation | page URL の JSON 配列 | 候補者は返信後に紐付け（評価時は空でよい） |

- **フォールバック（CLAUDE.md §14：Notion 書込が承認待ちで止まった時）**：Claude.ai は create が通らなくても、評価出力の末尾に下の「登録ブロック」をテキストで必ず残す。佐藤が後で貼る／Claude Code が **`tools/scout_eval.py`**（`NOTION_TOKEN` 直叩き＝§14 を回避。`create --from-block "…"` でこの登録ブロックをそのまま取り込める）で後追い upsert する。
  - **登録ブロック（評価時に分かる列・パイプ1行）**：
    `評価日｜媒体ID｜媒体｜現職カテゴリ｜総合点｜判定｜ペルソナ｜年代バンド｜推定heavy｜一言所感｜追加シグナル`

## 5. 取り扱い注意

- 進行中候補者の実名・現職名は伏せる（イニシャル＋業界カテゴリで記述）
- 機密度高の情報（年収上限・経営陣の NG 発言等）はファイル冒頭に
  `※Claude.ai 外への共有禁止` と明記
- 情報不足の項目は `[要確認：◯◯]` と残し、**推測で埋めない**（これは**キット生成時のクライアント／ポジションの事実**＝要件・経営陣スタンス等の話。**候補者評価のスパース profile 補完**は §4.4「経歴推定レイヤー」で `【推定】`＋確信度を明示したうえで推定してよい。両者を混同しない）
- 完了後、生成した5ドキュメントの Craft リンクを一覧表示し、Claude.ai プロジェクトへのアップロード手順を1行で添える

## 6. Claude.ai 側の運用

### 6.1 プロジェクト構成方針（会社単位を推奨）

**基本：会社単位の統合プロジェクト**を作り、同一企業の複数ポジションのスカウトキットをまとめて格納する。理由は以下：

- ヘッドハンティング業務では「同一企業の別ポジションへの横展開」が頻繁（CISO 不適合 → CIO 候補に振り替え等）
- 議事録・経営陣スタンス・企業概要などの**会社共通情報を1箇所で管理**できる
- 議事録更新がN箇所更新にならず運用負荷が低い

**構成判断基準：**

| 構成 | 向いているケース | 注意点 |
|---|---|---|
| 会社単位（推奨デフォルト） | 同一企業に複数ポジション、議事録更新が定期的、横展開判断したい | ファイル命名でポジション識別が必須 |
| ポジション単位（純化型） | 会社にポジション1つだけ／選考プロセスがポジション間で完全に別系統／1ポジションあたり評価候補 50件超 | 会社情報の重複管理コスト |

**会社単位プロジェクトのファイル構成例：**

```
{社名}スカウト評価プロジェクト/
├── 企業概要.md                       ← 会社共通
├── 経営陣スタンス.md                 ← 会社共通
├── 議事録_2025-Q4.md                 ← 会社共通（※社外共有禁止）
├── [CISO候補-{社コード}] ターゲット人物像.md
├── [CISO候補-{社コード}] 要件.md
├── [CISO候補-{社コード}] 過去推薦パターン.md
├── [CISO候補-{社コード}] 評価フレーム.md
├── [CIO候補-{社コード}] ターゲット人物像.md
├── [CIO候補-{社コード}] 要件.md
├── [CIO候補-{社コード}] 過去推薦パターン.md
└── [CIO候補-{社コード}] 評価フレーム.md
（※評価ログは project knowledge に置かず、Notion DB「スカウト評価ログ」へ直接 create＝§4.5。月次フラッシュは廃止）
```

- ポジション固有ファイル（Craft `{ポジション}-scout-kit/` 由来）は、Craft の doc タイトルが既に **`[{ポジション名}-{社コード}]` プレフィックス付き**（§4）なので、エクスポートしたファイル名のままアップロードする（手で付け直さない）
- 会社共通ファイル（経営陣スタンス・議事録・企業概要）はプレフィックスなし
- 機密度の高い議事録は冒頭に `※社外共有禁止` を明記

**会社単位用カスタム指示テンプレ：**

```
候補者評価リクエストを受けたら、最初に対象ポジション名を必ず確認すること。
評価では `[ポジション名-社コード]` プレフィックスのファイル群＋プレフィックスなしの会社共通ファイルのみを参照し、
他ポジションのファイルは混在参照しないこと。

評価は [ポジション名-社コード] 評価フレーム.md の5軸で採点する。
職務記述が薄い候補者（企業名・役職・在籍年月しか読めない LinkedIn 等）は、採点前に
評価フレームの「経歴推定レイヤー」で役職×在籍年数×企業フェーズからスコープを推定し
（【推定】＋確信度を明示、推定をスカウト本文に事実として書かない）、要件と突き合わせること。
評価が固まったら Notion DB「スカウト評価ログ」に1行 create する
（notion-create-pages / data source 437a2bad-94d6-4397-8dc4-1aa986a34c3d。
媒体ID＝BizReach会員ID または LinkedIn slug を title に、推定を多用したら 推定heavy=__YES__）。
Notion 書込が通らない時のフォールバックとして、末尾に登録ブロック
（評価日｜媒体ID｜媒体｜現職カテゴリ｜総合点｜判定｜ペルソナ｜年代バンド｜推定heavy｜一言所感｜追加シグナル）
もテキストで残す。
```

### 6.2 初期セットアップ（1回）

**会社単位プロジェクト（推奨）の場合：**

1. Claude.ai でプロジェクト作成（例：「{社名}スカウト評価」）。既存プロジェクトがあれば新ポジション分を追加するだけで OK
2. Craft の `{ポジション}-scout-kit/` 配下の5ドキュメントを md エクスポート（Craft アプリの「Share → Export as Markdown」）。doc タイトルが既に `[{ポジション名}-{社コード}]` 付き（§4）なので、ファイル名もそのプレフィックス付きで出る
3. エクスポートしたファイル名のままアップロード（プレフィックスの付け直しは不要。§4 で焼き込み済み）
4. 会社共通ファイル（`経営陣スタンス.md` / `企業概要.md` / 議事録）は初回のみアップロード（既存の場合は更新差分のみ差し替え）
5. プロジェクトのカスタム指示に 6.1 の「会社単位用カスタム指示テンプレ」を記載

**ポジション単位プロジェクトの場合：**

1. Claude.ai でプロジェクト作成（例：「{社名}_{ポジション}_スカウト評価」）
2. Craft の `{ポジション}-scout-kit/` 配下の5ドキュメントをそのままアップロード（doc 名に `[{ポジション名}-{社コード}]` が付くが、純化型では識別上は無くても可。付いていても問題ない）
3. プロジェクトのカスタム指示に以下を記載：
   ```
   候補者情報が貼られたら、評価フレーム.md の5軸で採点する。
   職務記述が薄い場合は先に「経歴推定レイヤー」で役職×在籍年数×フェーズからスコープを推定
   （【推定】＋確信度を明示）してから採点する。評価が固まったら Notion DB「スカウト評価ログ」に
   1行 create（data source 437a2bad-94d6-4397-8dc4-1aa986a34c3d、媒体ID を title に）。
   書込が通らない時のみ末尾に登録ブロックをテキストで残す。
   ```

### 6.3 日々の評価

1. 新規チャットで候補者プロフィール（BizReach / LinkedIn 等）を貼る
2. **会社単位プロジェクトの場合は対象ポジション名を冒頭で明示**（例：「[CISO候補] の評価をお願い」）
3. Claude が評価フレームの5軸で採点（職務記述が薄ければ経歴推定レイヤーを先に適用）し、**Notion DB「スカウト評価ログ」に1行 create** する（§4.5）。手ペーストは不要
4. Notion 書込が承認待ちで止まった場合（CLAUDE.md §14）のみ、Claude が出力末尾に残す「登録ブロック」を退避（佐藤が後で貼る／Claude Code が `tools/scout_eval.py create --from-block "…"` で取り込み）

### 6.4 月次リセット（廃止）

旧運用（Claude.ai の live `評価履歴.md` を月次で Craft `scout_log_{YYYY-MM}.md` へ流す）は**不要になった**。評価ログは Notion DB に貯まり続けるのでアーカイブを兼ねる。期間で見たい時は DB のビューを `評価日` で絞り、深掘り分析が要る時だけ DB を Google スプレッドシートへエクスポートする。

### 6.5 移行トリガー（＋アウトカム書き戻し）

- 候補者から返信が来た時点で、Claude Code 側で candidate md を新規作成し以降は正本連動運用へ移行する
- Claude.ai プロジェクトは**未接触プール専用**と割り切る

**アウトカム書き戻し（評価履歴を学習資産にする要）：** 返信→SF登録→Notion 自動連携→Claude Code でプロファイル作成、という導線に相乗りして、評価ログへ結果を書き戻す。専用の追記作業を作らず、プロファイル作成時の1ステップとして実行する（手順は `agents/candidate-profile.md` §5.2 に組み込み済み）。

- **照合キー：** 評価ログの `媒体ID`（＝DB の title。BizReach＝会員ID／LinkedIn＝プロフィールURL の slug／その他＝媒体の安定ID。§4.5 参照）。**媒体ID は SF Contact に保存されている**（BizReach 会員ID・LinkedIn URL 等。Notion 候補者には現状なし）。プロファイル作成時に SF を取得するので、SF の媒体IDで評価ログの `媒体ID`（title）と照合する。LinkedIn 由来は SF の LinkedIn URL の slug で突合する。SF にも該当IDが無い場合のみ氏名で曖昧照合（コスト・誤りが増えるので非推奨）
- **書き戻し先：** 正本は **Notion DB「スカウト評価ログ」**（data source `437a2bad-94d6-4397-8dc4-1aa986a34c3d`）。Claude Code が媒体ID（title）で該当行を検索し、`notion-update-page` で更新する（MCP 書込が止まれば `tools/scout_eval.py backfill --media-id … --reply-date … --result …` で。CLAUDE.md §14）。照合は媒体ID＋直近2〜3ヶ月（`評価日`）に絞る
- **更新内容：** 該当行の `返信日｜結果（面談/推薦/通過/見送り/辞退）｜SF id`（＋判明していれば `候補者` relation）を更新（返信そのものは返信日の有無で表す）
- **コスト：** プロファイル作成で既に SF を取得しているため追加トークンは小さい。肝はトークンではなく媒体IDによる行の特定

### 6.6 複数社・複数ポジション横断のルーティング運用（比較マップ）

「複数の注力ポジション（別社含む）を1つの claude.ai プロジェクトに入れ、貼られた候補者を**最も興味を持ちそうな1件へ振り分けて**、その案件にミートしたスカウトを書く」用途では、6.1 の会社単位プロジェクトとは構成を2点変える。

- **会社共通層は「比較マップ」1枚に畳む（会社概要 ×N を作らない）。** 各社の会社概要と差別化軸を**1枚の `注力ポジション比較マップ`** に集約する。これは単一企業の情報ではなく**会社・ポジションをまたいで比べるルーティング資産**なので、置き場所も1つにする。
  - 正本：Craft `work > 10_Recruitment > 12_Client｜企業` 直下の **`注力ポジション比較マップ`**（2026-06-17 新設。rootBlockId `620dff6e-b31a-9d63-81ab-dd61cfabe7dd`）。
  - 構成：①横並び比較テーブル（軸が行・ポジションが列／社コード・事業ドメイン・フェーズ・カルチャー・年収・組織体制・英語要否・必須/足切り・刺さる像・鮮度）②ルーティング軸（どの軸の順で振り分けるか）③各ポジション詳細（会社概要＋刺さる像＋足切り）④メンテナンス。
  - 各社 `{社名}.md` は正本のまま触らない。マップはそこから**差別化に効く最小限だけを薄く写した合成ビュー**で、詳細要件・経営陣は scout-kit が持つ（層を分け、二重管理を避ける）。
- **カスタム指示は 6.1 と逆向き（混在参照を許す）。** 「貼られた候補者を全ポジションの評価フレームで採点し、フィット×本人が興味を持ちそうな度合いで1〜2件にランキング（確信度付き）→ 最上位案件のスカウトまとめページの個別化スロットで下書き → 評価ログに1行 create」とする。6.1 の「他ポジション混在参照禁止」は単社純化型の指示なので、ルーティング用途では採用しない。
- **プロジェクトに入れるファイル**：scout-kit 5点（プレフィックス付き・×N）＋ スカウトまとめページ（×N）＋ **比較マップ1枚**。ポジション md と `{社名}.md` 全文は入れない（scout-kit と重複し肥大・機密拡散になる）。

**メンテナンス（更新は1箇所）：** ポジションが増えたら比較マップにテーブル1列＋詳細1ブロックを足すだけ（各社 md は触らない）。会社概要が実質的に変わったらマップの該当セルと鮮度（🟢🟡🔴＝各社 md の最終更新日基準）だけ更新する。更新後は md エクスポートで claude.ai 側の該当ファイルを差し替える。

## 7. 差分更新モード

既存の Craft `{社名}/positions/{ポジション}-scout-kit/` がある状態で「diff 形式で更新」と指示された場合：

1. 既存5ドキュメントを `blocks get` で取得
2. Craft / Notion / 1on1 / SF を再 fetch し、新しい情報・変化点を抽出
3. 各ドキュメントに対し、変更箇所のみを **before/after の diff ブロック** で提示
4. ユーザー承認後に `blocks update` / `blocks add` で反映
5. 反映後、Claude.ai プロジェクト側で「該当ドキュメントを Craft から再エクスポートして差し替えが必要」と1行アナウンス
