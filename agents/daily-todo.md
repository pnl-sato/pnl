# 朝のTODO生成ルーティン（その日のタスク＋スカウトノルマ）

毎朝の新セッションで「今日のタスクをリストして」と言われた時の手順。通常のToDoに加え、**注力ポジションのスカウト送信ノルマ（今日の通数）を自動算出**し、さらに**他メンバーのカレンダーから佐藤が同席すると良さそうな打ち合わせをスキャンして含める**（§4）。

> 読む正本：Notion ToDo DB（`collection://2257d017-b6a0-8026-867c-000bb0969507`）＋前日の DailyLog（Craft `15_DailyLog｜業務ログ`）＋本手順のスカウト算出＋他メンバーのGoogleカレンダー（§4）。トークン節約のためプロファイルの全文展開はしない（CLAUDE.md §13）。

---

## 1. 通常のToDoリスト化

1. Notion ToDo DB を query（未完了・優先度順、`[佐藤]`/`[Claude]`、relation 文脈）。**取得は `python3 tools/notion_active_todos.py --work`**（私用カテゴリを除外する exclude 方式。`--category "Pole&Line"` の include 方式は Category 空欄の業務タスクを取りこぼすため使わない＝CLAUDE.md §10）。`[佐藤]`/`[Claude]` のプレフィックスやスカウト関連はフィルタしないので、両担当・空欄業務とも一覧に出る。
2. 前日 DailyLog で積み残し・申し送りを拾う。
3. 優先度順に提示（年内売上の刈り取り＝面談・買い込み・クロージングを上位に。スカウトは下記ノルマとして別立て）。

## 2. スカウト送信ノルマの算出（注力ポジション）

1. **注力セグメントを特定（正本＝Notion ポジションDB の `注力` フラグ）**：何が注力かは **Notion ポジションDB の `注力=YES` が唯一の正本**。**この doc に固定の注力リストは持たない**（散文に列挙すると Notion と drift するため＝2026-06-17 方針）。タスク洗い出しでは **`python3 tools/scout_snapshot.py` の「注力=YES セグメント」セクションをそのまま注力として扱う**（snapshot は スカウトDB `使用中=YES` × ポジションの `注力` rollup をライブ解決して出す＝`scout_snapshot.py` line 80。媒体は集約して1セグメント＝ポジション×文面版）。
   - **注力を変えるときは、この doc ではなく Notion を更新する**：1on1 等で注力ポジションを足す／外すと決めたら、**Notion ポジションDB の当該ポジションの `注力` フラグ**（必要に応じてスカウトDB の `使用中`）を更新する。次回の朝ルーティン・snapshot に自動反映され、doc 改訂は不要。
   - 注力の変遷の経緯（なぜ入れた／外したか）は **1on1議事録（CLAUDE.md §9）と Notion** が正本に残す。例：2026-06-12 に法務・社長候補・CISO・Gifty営業を注力から外した、等の判断ログは議事録側へ。
2. **送信実績を取得**：**`python3 tools/scout_snapshot.py` を実行**。各セグメントの **今日／今週（月曜起点）／今月／累計／返信率**が出る（注力=YES が上）。データ源は **Notion「スカウト送信ログ」DB**（1行＝日付・スカウト文・通数）で、日付で解決するので**週初スナップショットは不要**。媒体別に分散した行はテンプレ名で名寄せ済。再加工は `--json`。
3. **今週送信済を読む**：snapshot の **「今週」列**をそのまま使う（差分計算・ベースライン不要）。注力合計も先頭に出る。
4. **週目標を決める（週100"基準"・等priority・カレンダー連動の変動枠）**：
   - **週100通は「佐藤の通常週capacityの総量」＝ポジション単位ではなく注力セグメント全体の合計**。注力が複数あっても「各セグメント週100」ではない（誤解しやすいので明示＝2026-06-16）。
   - **注力セグメントは等priority（同じ温度感）で並べ、"その週に打てる総量"を均等に山分け**する。注力2軸なら通常週で各50目安。面談が少なく空いた週は各50超もOK、面談で埋まった週は両セグメントとも一緒にしぼむ（**下限0・繰越は強制しない**）。
   - **総量はカレンダー連動の変動値**：面談・打合せが入るほど週総量・各セグメント枠とも自動で下げる（固定100に縛らない＝佐藤の実態「面談が増えると打てない」を反映）。実際の重み付けは §6 の日次算出が担う。
   - 新しい注力ポジションを足したら、**総量を上乗せするのではなく配分を割り直す**のが基本（capacityを上げる判断は別途。上限〜120フレックスの範囲で）。
   - フレックス：週次トラッキング表の **第1層EV合計が年末ゴール（あと3〜4決定）に遅れていれば上限寄り（〜120）**、順調なら100基準、余裕あれば抑制可。
   - §10 の返信率判定で **`3%未満＝止め` のセグメントがあれば、その配分を生きているセグメントへ寄せる**（死んだ文面に量を入れない）。
5. **カレンダー確認**：Google カレンダーで今日＋今週残りの空き状況を取得。
6. **今日の通数（カレンダー重み付け）**：`(週目標 − 今週送信済) ÷ 今週の残り稼働日` を、カレンダーの空きで重み付け（埋まった日は減らし、空いた日へ寄せる）。下限0・現実的上限 〜30/日。セグメント別に内訳。
   - カレンダー connector が当日無効なら**平日均等（週目標÷残平日）にフォールバック**し、その旨を明記。
7. **送信したら都度ログに記録**：その日に打った分は **`python3 tools/scout_log.py --template "<テンプレ名の一部>" --count N [--media Linkedin]`**（曖昧なら `--scout-id <page_id>`、一覧は `--list`）で記録する。**スカウトDB.送信数 は手で増やさない**（凍結済。累計は `送信数（ログ集計）` rollup が自動算出）。DailyLog への累積スナップショットはもう不要（ログDBが正本）。
   - **★ハイブリッド運用（2026-06-15〜）：日々のスカウトを「スカウト評価ログ」に 1候補=1行（判定・送信日・媒体・スカウト文）で残す運用なら、`scout_log.py` を手打ちする代わりに `python3 tools/scout_eval_to_log.py --date today` を流す。** 評価ログの**「送信日が入っている行＝送信済み」**（判定B以上は送信デフォルトで送信日を入れる／Cは送った場合のみ／Dは空）を 日付×スカウト文 で集計し、送信ログへ**冪等に**upsert する（メモ `eval-agg` マーカーで二重計上しない）。これで「日々は評価ログで自然に管理」「週次・累計・返信率は送信ログ＝`scout_snapshot.py`」を二重入力なしで両立できる。評価ログ行に**スカウト文が未紐付けだと集計されない**ので、その場合は `--assign-template "<テンプレ名 or page_id>"` で対象日の送信行に後付けしてから流す（評価ログ側も埋まり次回以降は自動で乗る）。評価を通さない一括送信は従来どおり `scout_log.py` で直接1行。

## 3. ToDoへの記載フォーマット（例）

```
[佐藤] スカウト送信 本日 24通（今週 32/100｜役員直下v2 16・toC 8）
   根拠：残68通 ÷ 残3日、本日は午後に空き多め／前日実績 18通
```

- 個別化スロットの下書きが要るなら `[Claude] スカウト下書き {N}通分（{セグメント}）` も併記。
- セグメントを止め/差し替えた場合は理由を1行（例：「toCは返信率2%で停止、役員直下に再配分」）。

## 4. 他メンバーの予定から「同席候補」をスキャン（2026-06 佐藤指示）

朝ToDoを出すとき、他メンバーのカレンダーを横断で見て、**佐藤が同席すると価値が出そうな打ち合わせ**を1〜数件ピックして併記する。基準は固定せず、**毎朝候補を出して佐藤の反応で絞り込む**運用（「これは見たい／これは要らない」を蓄積して精度を上げる）。下記は初期フィルタ。

1. **対象カレンダー（社内メンバー）：** `masakisaito@pnl.co.jp`（代表）／`yamada@pnl.co.jp`／`yusukefuruichi@pnl.co.jp`／`r.shimizu@pnl.co.jp`／`satoshi_matsumura@pnl.co.jp`。`mcp__Google_Calendar__list_events` を各IDで叩く（calendar ID 一覧は `mcp__Google_Calendar__list_calendars`）。
   - **対象外（2026-06 佐藤指示）：** 杉林（`ksugibayashi@pnl.co.jp`）・櫻井（`tsakurai@pnl.co.jp`）・河内（`suzuka.kawauchi@pnl.co.jp`）はスキャン対象に含めない。
2. **スキャン窓：** 平日は当日＋翌営業日を基本、**月曜は1週間先まで**広げて先回りする（直前だと予定を空けられないため）。
3. **拾う（＝同席価値が高い）：**
   - 新規クライアントの**初回打ち合わせ／キックオフ**（シニア同席で関係構築・受注角度が上がる）。
   - **大型リテーナーの定例・戦略MTG**（例：東京ガス高度DX人材・JDD など、注力領域に重なるもの）。
   - **ハイレイヤー候補者の最終面接・重要面接**（CxO・部長・経営企画・人事・セキュリティ）。
   - 佐藤の**注力領域（CxO／部長／経営企画／人事／セキュリティ）にフィットするクライアント面談**。
4. **落とす（＝ノイズ）：**
   - 社内1on1・全体MTG・私用/育児・健康診断などの個人予定。
   - スカウト媒体の運用打合せや媒体担当者レベルのMTG（実務担当で完結するもの）。
   - 官公庁案件（総務省・各県＝領域が別。佐藤が明示的に見たいと言わない限り出さない）。
   - **佐藤が既に参加予定のもの**（プライマリ／`03_面接・面談` に同じ予定があれば重複除外）。
5. **出力フォーマット：** 候補ごとに `日時／主催メンバー／打ち合わせ名／同席価値の一言／佐藤の空き（🟢空き・⚠️要調整・✕重複）` を1行で。0件なら「同席候補なし」と明示。多くても3〜4件に絞る（朝の俯瞰を重くしない）。
6. **精度の育て方：** 佐藤が「これは見たい／要らない」と反応したら、その傾向（どのクライアント・領域・メンバーを優先/除外するか）を本§の初期フィルタに追記して更新する。注力クライアント／領域の上位方針は `agents/work-approach.md`、注力ポジションは Notion ポジションDB `注力=YES`（§2）とも整合させる。

## 5. 注意

- 新規スカウトはリードタイムで **2027寄与が大きい**。年内決定の主力は既存パイプラインの刈り取り（週次トラッキング第1層）。スカウトノルマは「2027エンジン＋年内上振れ＋capacity消化」の位置づけで、**刈り取りToDoを圧迫しない範囲**で出す。
- **注力セグメントは「固定の週N通」ではなく等priority・カレンダー連動の変動枠**（2026-06-16 確定）。複数の注力ポジションを「同じ温度感」で回す＝各々に固定100を積むのではなく、その週に打てる総量を**均等配分**し、面談が増えた週は全体を一緒にしぼめる（下限0・繰越強制なし）。「注力を1つ足したら自動で+100」ではない＝総量は capacity とカレンダーで決まる変動値。媒体判定は媒体IDの形で行う（`BU+数字`=Bizreach／人名・slug=LinkedIn、同日両媒体あり得る＝`scout_eval.py` の `_media_auto` と同基準）。
- **送信の正本は Notion「スカウト送信ログ」DB**（2026-06-12 移行）。1行＝日付・スカウト文(relation)・通数・種別(送信/繰越)。`送信数（ログ集計）`=ログ合算rollup、`返信率`=返信数÷ログ集計。**旧 `送信数`（手動累計カウンタ）は 6/12 で凍結**＝以後は触らず、ログに dated 行を足す運用（`tools/scout_log.py`）。種別=繰越 は 6/12 時点の旧累計の移行行。
- 返信は当面スカウトDB `返信数` のまま（低頻度）。文面はポジション直下のまとめページ、PDCAの判定ルールは `agents/scout.md` §10。
