# 朝のTODO生成ルーティン（その日のタスク＋スカウトノルマ）

毎朝の新セッションで「今日のタスクをリストして」と言われた時の手順。通常のToDoに加え、**注力ポジションのスカウト送信ノルマ（今日の通数）を自動算出して含める**。

> 読む正本：Notion ToDo DB（`collection://2257d017-b6a0-8026-867c-000bb0969507`）＋前日の DailyLog（Craft `15_DailyLog｜業務ログ`）＋本手順のスカウト算出。トークン節約のためプロファイルの全文展開はしない（CLAUDE.md §13）。

---

## 1. 通常のToDoリスト化

1. Notion ToDo DB を query（未完了・優先度順、`[佐藤]`/`[Claude]`、relation 文脈）。
2. 前日 DailyLog で積み残し・申し送りを拾う。
3. 優先度順に提示（年内売上の刈り取り＝面談・買い込み・クロージングを上位に。スカウトは下記ノルマとして別立て）。

## 2. スカウト送信ノルマの算出（注力ポジション）

1. **注力セグメントを特定**：ポジションDB `注力=YES` 配下の、スカウトDB `使用中=YES` テンプレ（媒体は集約して1セグメント＝ポジション×文面版）。現状の**注力2軸**＝**①ギフティ事業開発系**（役員直下v1/v2・PMI推進・toC・公共行政）**②ギフティ人事**（グループ人事・採用 HD化／CHRO候補）。ポジションDBの `注力` フラグで自動追従（2026-06-12 に法務・社長候補・CISO・Gifty営業を注力から除外＝法務撤退・スカウト2軸化）。
2. **送信実績を取得**：**`python3 tools/scout_snapshot.py` を実行**。各セグメントの **今日／今週（月曜起点）／今月／累計／返信率**が出る（注力=YES が上）。データ源は **Notion「スカウト送信ログ」DB**（1行＝日付・スカウト文・通数）で、日付で解決するので**週初スナップショットは不要**。媒体別に分散した行はテンプレ名で名寄せ済。再加工は `--json`。
3. **今週送信済を読む**：snapshot の **「今週」列**をそのまま使う（差分計算・ベースライン不要）。注力合計も先頭に出る。
4. **週目標を決める（週100基準・自動フレックス）**：
   - 基準は **週100通／注力2セグメント**（佐藤の capacity）。
   - 週次トラッキング表の **第1層EV合計が年末ゴール（あと3〜4決定）に遅れていれば上限寄り（〜120）**、順調なら100、余裕あれば抑制可。
   - §10 の返信率判定で **`3%未満＝止め` のセグメントがあれば、その配分を生きているセグメントへ寄せる**（死んだ文面に量を入れない）。
5. **カレンダー確認**：Google カレンダーで今日＋今週残りの空き状況を取得。
6. **今日の通数（カレンダー重み付け）**：`(週目標 − 今週送信済) ÷ 今週の残り稼働日` を、カレンダーの空きで重み付け（埋まった日は減らし、空いた日へ寄せる）。下限0・現実的上限 〜30/日。セグメント別に内訳。
   - カレンダー connector が当日無効なら**平日均等（週目標÷残平日）にフォールバック**し、その旨を明記。
7. **送信したら都度ログに記録**：その日に打った分は **`python3 tools/scout_log.py --template "<テンプレ名の一部>" --count N [--media Linkedin]`**（曖昧なら `--scout-id <page_id>`、一覧は `--list`）で記録する。**スカウトDB.送信数 は手で増やさない**（凍結済。累計は `送信数（ログ集計）` rollup が自動算出）。DailyLog への累積スナップショットはもう不要（ログDBが正本）。

## 3. ToDoへの記載フォーマット（例）

```
[佐藤] スカウト送信 本日 24通（今週 32/100｜役員直下v2 16・toC 8）
   根拠：残68通 ÷ 残3日、本日は午後に空き多め／前日実績 18通
```

- 個別化スロットの下書きが要るなら `[Claude] スカウト下書き {N}通分（{セグメント}）` も併記。
- セグメントを止め/差し替えた場合は理由を1行（例：「toCは返信率2%で停止、役員直下に再配分」）。

## 4. 注意

- 新規スカウトはリードタイムで **2027寄与が大きい**。年内決定の主力は既存パイプラインの刈り取り（週次トラッキング第1層）。スカウトノルマは「2027エンジン＋年内上振れ＋capacity消化」の位置づけで、**刈り取りToDoを圧迫しない範囲**で出す。
- **送信の正本は Notion「スカウト送信ログ」DB**（2026-06-12 移行）。1行＝日付・スカウト文(relation)・通数・種別(送信/繰越)。`送信数（ログ集計）`=ログ合算rollup、`返信率`=返信数÷ログ集計。**旧 `送信数`（手動累計カウンタ）は 6/12 で凍結**＝以後は触らず、ログに dated 行を足す運用（`tools/scout_log.py`）。種別=繰越 は 6/12 時点の旧累計の移行行。
- 返信は当面スカウトDB `返信数` のまま（低頻度）。文面はポジション直下のまとめページ、PDCAの判定ルールは `agents/scout.md` §10。
