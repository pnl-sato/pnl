# クライアント／ポジションプロファイルエージェント

クライアント企業ごとに **Craft** にハイブリッド構造のドキュメントを置き、Notion + Salesforce + 打ち合わせメモ + 過去推薦履歴を集約する。マッチング評価の基礎データとなる。

`/client {社名}`、`/position {社名} {ポジション}`、`/match {候補者}` などのスキルから利用される。

---

## 0. 重要：プライバシー

- クライアント／ポジションプロファイルは **Craft 上にのみ保存**する。git・GitHub・他クラウドへの転載は禁止。
- クライアント側担当者の氏名・連絡先は Craft 内で扱うが、対外共有時は本人同意確認。
- 候補者プロファイル（`agents/candidate-profile.md`）とは別ファイルで管理するが、マッチング時には両者を参照する。

---

## 1. 保存先（Craft）

- **トップフォルダ:** `work > 10_Recruitment > 12_Client｜企業`
  - Craft folder ID: `F41A3C28-9B9B-47C7-BAC1-0D8431AF73A9`
- **各社のサブフォルダ:** `{社名}/`（例: `ギフティ/`）
- **その配下：**
  - `{社名}.md` — クライアント md（会社プロファイル＋関係性＋全体パターン）
  - `positions/` — サブフォルダ
    - `{ポジション名}.md` — ポジション md（要件・選考履歴・現パイプライン）

例：
```
12_Client｜企業/
├─ ギフティ/
│  ├─ ギフティ.md
│  └─ positions/
│     ├─ CISO候補.md
│     └─ 企業法務.md
├─ NodeX/
│  ├─ ...
```

既存7件の汎用ドキュメント（`Client List` `JD` `クライアント` 等）はそのまま共存。新規にクライアント md を作る際はサブフォルダで分ける。

## 2. 起動モード

| モード | 判定条件 | 振る舞い |
|---|---|---|
| **読み込み** | 会話中でクライアント名／ポジション名が言及された | Craft フォルダを search → 該当ドキュメントを `blocks get --format markdown` で全文取得 |
| **初回生成** | **セクション 2.5 の多重照合をすべて通過し「未作成」と判定された場合のみ** | Notion + SF + 打ち合わせメモ + 過去推薦履歴 + **Google Drive `01_企業情報/{社名} #{社コード}/` 配下の関連ファイル**（セクション 4.4 参照）から横断収集 → 不足分は **企業 Web / Web 検索**（セクション 4.5）で補完して Craft に新規作成 |
| **対話追記** | ユーザーが素材（打ち合わせメモ・採用要件メモ等）を貼り付け | 該当セクションに `blocks add` で要約追記 |
| **同期更新** | 「最新に同期」「ステータス反映」等の指示 | Notion・SF を再取得して該当セクションを更新 |
| **マッチング評価** | `/match` 系コマンド or「合いそうな案件は？」 | 候補者 md ＋ 全注力ポジション md を読み比較・スコア付け |

> **1on1報告フィードへの追記（対話追記・同期更新で報告価値のある動きがあれば）：** 要件・年収条件の変更、パイプライン進捗、打ち合わせ所感、新規打診の開始など代表に共有すべき差分が出たら、`agents/one-on-one.md` §2 に従い Craft `work > 20_1on1` の 1on1 報告フィードへ「日付＋概要」を1行追記する（軽微な同期は不要）。

## 2.4 クライアント判定（候補者の在籍企業と切り分ける）★初回生成・夜間 backfill で必須

SF の取引先(Account)には、**P&L のクライアント**だけでなく**候補者の在籍企業（職歴・現職の会社）**も大量に登録されている（2026-06 時点で Account 約14,000件のうち、実クライアントは171件のみ）。在籍企業をクライアントとして Craft に md 化してしまう事故（過去の夜間 backfill で多発）を防ぐため、Account を「クライアント」として扱う前に必ず以下で判定する。

**クライアントの判定条件（唯一の基準）：** `Contract_Status__c = '締結済み'`（契約状況＝締結済み）の Account **のみ**がクライアント。

| `Contract_Status__c`（契約状況） | 意味 | 扱い |
|---|---|---|
| `締結済み` | 契約締結済みの現クライアント | ✅ クライアント（md 化対象） |
| `null`（未設定） | 大多数は**候補者の在籍企業**（職歴の会社） | ❌ 非クライアント・md 化しない |
| `未締結` | 未契約 | ❌ 非クライアント・md 化しない |
| `締結対応中` | 契約手続き中（まだクライアントではない） | ❌ 非クライアント（締結後に対象化） |
| `終了` | 取引終了の旧クライアント（クローズ系） | ❌ 除外 |

SOQL 例：

```sql
SELECT Id, Name, Contract_Status__c, Notion_Page_ID__c, ATS_URL__c, OwnerId
FROM Account
WHERE Contract_Status__c = '締結済み'
```

> 候補者プロファイル側で在籍企業（職歴の会社情報）を参照する目的で Account を fetch するのは可。ただしそれを `12_Client｜企業` 配下に md 化してはならない。クライアント md 化は `Contract_Status__c = '締結済み'` を通過した Account に限る。

## 2.5 既存判定の多重照合（重複作成防止）★夜間 backfill では必須

社名の表記揺れ（「株式会社X」「X」「X Inc.」「英語名」）や Craft search のスコープ齟齬で**既存クライアントの薄い重複コピーを作ってしまう事故**が発生したため、初回生成モードに入る前に以下の多重照合を必ず実施する。

### 既存判定フロー（順に試行、いずれか1つでも該当すれば「作成済み」と確定）

1. **SF Account ID 照合（最強・絶対条件）**
   - Salesforce Account の Id（例：`0015h00000aCAqwAAG`）を取得
   - Craft 内を `mcp__craft__craft_read` の `search "Account {Id後ろ6文字}"` で grep（例：`search "Account aCAqwAAG"`）
   - 既存クライアント md の冒頭にある `**Salesforce:** Account 0015h00000aCAqwAAG` 行に必ずヒットするので、ヒットがあれば即「作成済み」と判定して**スキップ**

2. **Notion 企業 Page ID 照合**
   - SF Account に紐づく Notion Page URL（例：`notion.so/2ca7d017b6a08059a09cef70f1a64ec7`）を取得
   - Craft 内を `search` でその Page ID 後ろ8文字程度で grep
   - ヒットがあれば「作成済み」と判定

3. **`12_Client｜企業` 直下フォルダの完全一致＋配下文書チェック**
   - `folders list --filter "{社名}"` でトップ階層配下の同名フォルダがあるかチェック
   - 同名フォルダがあれば、`documents list --folder <ID>` で `{社名}.md` が配下にあるか確認
   - **空フォルダだけが残っているケースは「作成済み」扱い**（過去の手動削除で文書だけ消えた可能性）

4. すべて不一致 → **本当に未作成**と確定して初回生成モードへ進む

### 命名ルール（SF Account.Name 基準）★新規作成時は必須

クライアント md／フォルダの命名は **SF Account.Name を基準にした決定論的ルール**で固定する（2026-06 佐藤指示）。短縮名（通称）を人／AI が作ると run ごとにブレ、過去の重複事故（2026-05 マネーフォワード／NTTデータ／三井住友カード等で「株式会社X」「X株式会社」と短縮名フォルダが二重生成）の原因になった。判断の入らない機械的ルールにすることでブレを根絶する。重複判定の主キーはあくまで **SF Account ID（判定1）** であり、タイトルは「正本（SF）一致＋決定論的であること」を満たせばよい。

- **タイトル ＝ SF Account.Name から、最初の `/` または全角スペース（`　`）以降（別名・コード部分）だけを機械的に落とした文字列。** `株式会社`・`(株)`・`Inc.`・英語社名などは **落とさず SF のまま残す**。フォルダ名・doc タイトル・`{社名}.md` の3つをこの同一文字列で揃える。
  - `株式会社BuySell Technologies/ バイセルテクノロジーズ` → `株式会社BuySell Technologies`
  - `Japan Digital Design株式会社　JDD` → `Japan Digital Design株式会社`
  - `グーグル合同会社/ Google Japan/ Google Cloud Japan` → `グーグル合同会社`
  - `株式会社メドレー`（区切りなし）→ `株式会社メドレー`（そのまま）
- **作成直前の最終ガード：** 既存判定（判定1〜4、特に SF Account ID 照合）を通過していること。加えて `folders list` で `12_Client｜企業` 直下に**同一タイトルのフォルダ**が無いか確認し、あれば新規作成せず既存を正本にする。
- **既存フォルダ（2026-06 以前に短縮名で作成済み。BuySell Technologies・ギフティ等）はこのルールの遡及対象外**＝リネームしない。**新規作成分からのみ適用**する。重複判定は SF Account ID 主キーなので、短縮名の既存と SF 名の新規が二重生成される心配はない（同一 Account ID は判定1でヒットしてスキップされる）。
- 重複を見つけた場合の整理は MCP で完結できる：固有情報を正本へ統合 → 重複 doc は `documents delete --document <rootBlockId>`、空フォルダは `folders delete --folder <folderId>`（いずれも soft-delete、30日リカバリ可）。※doc タイトルのリネーム機能は MCP に無いため、タイトル不一致のみ残る場合は手動。

### 夜間 backfill 専用の追加チェック

- 5/29 03:30 のような自動ジョブで作成された薄いコピー（「特記事項なし」「（未取得）」が多発）は**正本ではない**。判定 1〜3 をすべて通過した後でも、生成前に最後の安全弁として：
  - `folders list` の `12_Client｜企業` 直下に**同名フォルダが既に複数存在**していたら、新規作成せずユーザーに確認
- 重複作成事故が起きた場合は `documents delete --document <rootBlockId>` で 30 日 soft-delete（リカバリ可能）

### 表記揺れ吸収のための補助 search クエリ

- `{社名}`（プレーン）
- `株式会社{社名}`
- `{社名}株式会社`
- `{英語社名}`（SF Account.Name_English__c があれば）
- `#{社コード}`（Google Drive と一貫）

これらを順に試して、いずれかでヒットすれば既存とみなす。

会社の共有 Gmail は別アカウントで Claude から直接見れないため、**個人 Gmail（sato-y@pnl.co.jp）への自動転送＋ラベル振り分け**で同期している。

### ラベル設計

- **個人 Gmail 由来：** プレフィックスなし。例：`GFT`（ギフティ）、`KHS`（カケハシ）
- **共有 Gmail 由来：** `SY/` 配下にネスト。例：
  - `SY/GFT` — ギフティ関連の共有メール（クライアントからの公式やり取り、ポジション情報など）
  - `SY/ATS-HERP` — HERP 系 ATS 通知（複数クライアント横断）
  - `SY/ATS-HRMOS` — HRMOS 系 ATS 通知
  - `SY/ATS-Talentio` 等

### 共有 Gmail 側のフィルタ Recipe

クライアントごと（注力クライアントのみ）：
```
Has the words: "{社名}" OR "{英語社名}"
→ 転送: sato-y@pnl.co.jp
→ ラベル: SY/{社コード}
```
※ Gmail の「Has the words」は subject + body + snippet を検索

ATS 共通フォールバック（全クライアント横断、7 ATS 系統を一括）：
```
from:(*@herp.cloud) OR from:(*@hrmos.co) OR from:(*@talentio.com)
  OR from:(*@greenhouse.io) OR from:(*@lever.co)
  OR from:(*@persona-ats.com) OR from:(*@jposting.net)
→ 転送: sato-y@pnl.co.jp
→ ラベル: SY/ATS（または ATS 種別ごとに分割可）
```

### 検索順序（マッチング評価・候補者プロファイル生成時）

1. **個人 Gmail（直接やり取り）：** `search_threads` with `{候補者メアド}` or `{候補者氏名}`
2. **共有 Gmail 由来（クライアント文脈）：** `search_threads` with `label:SY/{社コード} "{候補者姓}"`
3. **共有 Gmail 由来（ATS 通知）：** `search_threads` with `label:SY/ATS "{候補者姓}" "{社名}"` （ATS 通知は subject/body に **{ATS_ID}@{ATSドメイン}** 形式の sender で来る、候補者氏名は subject に含まれる前提）

### 重要：sender アドレスベースでの候補者識別はしない

ATS 通知の sender は `{ランダム候補者ID}@{ATSドメイン}` 形式で、毎回違うため。**識別は subject（企業名 + 候補者名）または body 内容**で行う。

### クライアント名・候補者名の判別パターン（共有 Gmail）

| 種別 | 判別 | 検索クエリ例 |
|---|---|---|
| 候補者メール | subject に企業名 + 候補者名 | `label:SY/GFT "八木 俊輔"` |
| ポジション開閉（subject 型） | subject に企業名 | `label:SY/GFT (closed OR open OR クローズ OR オープン)` |
| ポジション開閉（body 型） | 本文に企業名 | `label:SY/ATS "ギフティ"` |

## 4.4 Google Drive 検索戦略（企業情報フォルダ）

クライアントごとの採用要件メモ・JD 原本・打ち合わせ資料・ピッチ資料などは、Google Drive の **`01_企業情報` フォルダ**（folder ID: `1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx`）配下に、社ごとのサブフォルダで集約されている。

### フォルダ命名規約

サブフォルダ名は **`{社名} #{社コード}`** 形式。社コードは Gmail `SY/` ラベルと同じもの。

例：
- `ギフティ #GFT`
- `MIXI #MXI`
- `BuySell Technologies #BST`
- `テイラー #TYL`

（社コード未付与の旧フォルダ：`DMM`, `テイラー`（重複）など。コード入りの方を正本扱いとする）

### クライアント md 作成・更新時のフロー

1. **社フォルダを特定**：`search_files` で `parentId = '1LP3St--FiVzlpQxJlqD3bjOj1ju5Z4Gx' and mimeType = 'application/vnd.google-apps.folder' and title contains '#{社コード}'` を実行
   - ヒットしなければコードなしの社名一致（`title contains '{社名}'`）でフォールバック
2. **配下のファイル一覧を取得**：見つかった社フォルダの ID を `parentId` に指定し `search_files`（`modifiedTime` 降順で目視判定）
3. **関連ファイルを選定**：タイトル・更新日・mimeType から、採用要件メモ・JD・打ち合わせ議事録・ピッチ資料など、クライアント md の根拠になり得るものを Claude が選ぶ
4. **本文取得**：`read_file_content` で内容を展開。Google Docs / Sheets / Slides も対応
5. **クライアント md に反映**：
   - 採用要件メモ → 該当ポジション md の「ポジション要件」「JD 抜粋」
   - 打ち合わせ議事録 → クライアント md の「打ち合わせメモ要約」「P&L 経由の推薦傾向」
   - ピッチ資料 → クライアント md の「会社概要」「経営陣・キーパーソン」
6. **クライアント md の Sources 行に Drive 参照を追加**：`Google Drive ✓` と社フォルダの viewUrl を載せる

### 注意

- サブフォルダ配下にさらに階層がある場合、必要に応じて再帰的に list する（ただし関連性の低い深い階層は読まない）
- Drive のファイルは更新日が新しいものを優先（同名の古いバージョンが残っていることがある）
- 添付の Word/Excel/PDF も `read_file_content` で読めるが、大容量は metadata だけ確認して必要部分だけ読む

## 4.5 Web 検索戦略（企業 Web サイト／Web 検索）

公式 Web サイトと一般 Web 検索は、**内部ソース（Notion / SF / Slack / Gmail / Drive）で埋まらない事項を補完する外部ソース**。鮮度の高い情報（直近ニュース）や、Drive にピッチ資料がない企業の会社概要を埋めるのに使う。

### 使い分け

| ツール | 使う場面 | 例 |
|---|---|---|
| **WebFetch** | 見るべき URL が分かっている（公式の会社概要・採用ページ・IR・テックブログ） | `https://{社名}.co.jp/about/` を取得して会社概要・経営陣を埋める |
| **WebSearch** | URL 不明、または直近ニュースを探す | `{社名} 資金調達`、`{社名} 新規事業 2026`、`{代表者名} インタビュー` |

### 優先して見るページ（WebFetch）

1. 会社概要 / About（`/about` `/company`）→ クライアント md「1. 会社概要」
2. 経営陣 / Member（`/member` `/team`）→「2. 経営陣・キーパーソン」
3. 採用ページ / Careers → ポジション md の文脈補強（JD の正本は SF / Drive）
4. IR（上場企業）→ 売上・成長率・成長戦略
5. テックブログ / プレスリリース → カルチャー・技術スタック・直近の動き

### 環境上の注意（実測）

- **WebSearch は安定して動作する。** まず WebSearch で当たりを付け、必要なら結果 URL を WebFetch するのが堅い。
- **WebFetch は企業公式サイトで 403（bot 対策）を返すことがある**（例：giftee.co.jp）。その場合は無理に踏まず、(a) WebSearch のスニペット、(b) アグリゲータ（Baseconnect・各種転職メディア等）、(c) IR・プレスリリースのキャッシュ等で代替する。
- WebFetch は HTTP→HTTPS に昇格し、クロスホストのリダイレクトは追わず返すので、返ってきた URL で再度呼ぶ。
- ネットワークアクセスは本リモート環境のポリシーに従う。ポリシーによっては特定ドメインしか踏めない / 全面不可の場合がある。踏めなければ「Web ソースは今回参照不可」と md に明記する。

### 裏取り・記録ルール（必須）

- Web 由来の数値・事実（従業員数・売上・資金調達額など）は、**内部ソースと食い違う場合は内部ソースを優先**し、差分を「留意事項」に残す。
- クライアント md の Sources 行に `Web ✓（YYYY-MM-DD取得）` を追加し、本文中の Web 由来記述には取得元 URL を併記する。
- アグリゲータ（転職メディア等）の二次情報は鮮度・正確性が劣るため、公式・IR・一次報道を優先する。

## 5. 同期戦略（重要）

候補者数・ポジション数の規模を考慮し、**コスト×スピードのバランス**で2層運用：

| 区分 | 保管方法 | 同期方法 | マッチング時 |
|---|---|---|---|
| **注力 = YES** ポジション | Craft に md として常設 | `/position-sync` で手動同期 or 重要更新時に Claude が自動 | md を直接読む |
| **注力 = NO** ポジション | Craft 化しない | 都度 Notion / SF から fetch | マッチング時に Notion ポジション DB を query |

**根拠：**
- 注力ポジションは P&L 独自の知見が厚く、永続化のリターン大（20件程度）
- 非注力は数が多く（100件）、SF が真の最新ソース。md 化はコスパ悪い
- マッチング時は「注力 md 全件 + 必要なら非注力スナップショット」の二段取り

「注力 → 非注力」または「注力 → クローズ」フラグ変化時：md を Archive サブフォルダに移動。

## 7. マッチング評価（`/match` フロー）

### ポジション universe の定義（重要）

**まず Notion ポジション DB（`collection://1fb7d017-b6a0-8052-a7c9-000b1aa76cda`）で、P&L が実際に管理しているオープン案件を確認する**（注力フラグ・職種/職位・報酬レンジ・決定確度・契約形態が揃っており、意思決定に効く質の高い一次母集団になる）。そのうえで **SF Opportunity（ポジションの正本）** で母集団を補完し、`LimitedreleaseURL__c` の有無で2カテゴリに分類してマッチング評価する（Notion 未掲載の公開求人＝新規打診候補は SF 側で拾う）：

| カテゴリ | 判定 | 意味 | アクション |
|---|---|---|---|
| **🟢 即推薦可** | `LimitedreleaseURL__c IS NOT NULL` AND `StageName != クローズ系` | P&L が ATS 経由でアクセス可能（直接メール依頼も最終的にここに登録される） | 推薦状作成 → ATS / HERP Hire 提出 |
| **🟡 新規打診候補** | `LimitedreleaseURL__c IS NULL` AND `URL__c IS NOT NULL` AND `StageName != クローズ系` | 公開のみ（Bot 自動登録）、P&L 未契約 or ATS 未登録 | 「新規アプローチすべきクライアント」として人事窓口にコンタクト |

**Slack `#new_job3_new`** は Claude が直接読む対象ではない（人間向けの informational push、中身は SF Opportunity への link 集約）。

**Email 共有由来（`SY/` ラベル下）** は **context enrichment** ソース：
- 新ポジション発見には使わない（すべて SF に集約される）
- 過去のクライアントとのやり取り（判断理由・カルチャー上の懸念・年収交渉履歴）
- ATS 通知の詳細補足（推薦結果・選考フィードバック・面接調整）
- → 評価軸「過去パターン」「カルチャー」の根拠となる文脈情報

### 情報の鮮度を考慮する（重要）

ポジション要件・報酬レンジ・ステータス・パイプラインは動きが速く、古い情報で推薦すると「すでにクローズ」「要件が変わった」「枠が埋まった」といった空振りが起きる。マッチング評価では各ポジションの**最終更新日**を必ず取得し、鮮度を評価に織り込む。

**最終更新日の取得元（新しい方を採用）：**
- Notion ポジション DB の各ページの「最終更新日時」（`last_edited_time`）
- SF Opportunity の `LastModifiedDate`（要件・ステージの更新）／補助的に `LastActivityDate`
- Craft ポジション md の場合はヘッダの「ポジション最終更新」行（無ければ md 化以降に Notion/SF を再取得して更新）

**鮮度区分（`Today's date` を基準に経過日数で判定。日付判断は CLAUDE.md セクション6に従い毎回 `Today's date` を再参照）：**

| 鮮度 | 経過日数 | 扱い |
|---|---|---|
| 🟢 新鮮 | 〜30日 | そのまま評価に使ってよい |
| 🟡 要確認 | 31〜90日 | 評価に使うが、推薦前に最新ステータス・枠の有無を一次ソース（SF / ATS / クライアント窓口）で確認するよう併記 |
| 🔴 要再同期 | 90日超 | 情報が古い前提で扱う。推薦アクション前に再同期（SF/Notion 再取得 or クライアント確認）を必須とし、適合スコアには注記を付ける |

**評価への反映方針：**
- 鮮度は**適合度スコアそのものを直接下げる軸ではない**（古い＝不適合ではない）。あくまで「その評価がどれだけ信頼できるか」「推薦前にどこまで再確認が要るか」を示すリスク注記として扱う。
- 同程度の適合度が並んだ場合は、**鮮度の高い（最近動きのある）ポジションを優先**して上位に置く。クライアントが直近で動かしている＝採用意欲が高く決まりやすいシグナルでもある。
- 🔴 要再同期のポジションを上位に出す場合は、「情報が古いため要再確認」を懸念点の筆頭に明記する。

### 候補者起点（`/match {候補者姓}` または自然言語）

1. Craft から候補者 md を fetch
2. **まず Notion ポジション DB を query**（`collection://1fb7d017-b6a0-8052-a7c9-000b1aa76cda`、`ステータス = オープン`）→ P&L が実際に持つ案件・注力フラグ・職種/職位・報酬レンジ・決定確度・契約形態を把握（質の高い一次母集団）。候補者の職種/フェーズ適性で粗く絞る
3. **次に SF Opportunity を query**（`StageName != クローズ系`、Limit 100程度）→ 母集団を補完して universe 確定（Notion 未掲載の公開求人＝新規打診候補もここで拾う）。**この際、各ポジションの最終更新日（Notion `last_edited_time` / SF `LastModifiedDate`）も併せて取得し、鮮度区分（🟢/🟡/🔴）を付与する**（「情報の鮮度を考慮する」参照）
4. 各 Opportunity を 2 カテゴリに分類：
   - 🟢 = `LimitedreleaseURL__c IS NOT NULL`
   - 🟡 = `LimitedreleaseURL__c IS NULL`（公開のみ）
5. 注力 YES（Notion ポジション DB の `注力` フィールド）に該当するものは Craft 注力ポジション md でさらに深掘り
6. Email 共有由来を context として fetch：
   - 該当クライアントの `label:SY/{社コード}` 直近メール
   - `label:SY/ATS-*` の該当候補者氏名を含むスレッド
7. 各ポジションについて、以下5軸で構造化評価：
   - **要件マッチ度**（Must-have の充足度）
   - **カルチャーマッチ度**（本人志向 × クライアント文化、過去通底パターン参照）
   - **年収マッチ度**（本人希望 × ポジション提示レンジ）
   - **フェーズマッチ度**（本人のフェーズ適性 × 企業ステージ）
   - **過去パターン**（このクライアント／類似ポジションでの脱落パターンに照らした懸念点。Email 共有由来からも補強）
   - **鮮度**（最終更新日に基づく 🟢/🟡/🔴。スコア軸ではなくリスク注記。同点なら鮮度の高い方を優先、🔴 は要再確認を明記）
8. **🟢 と 🟡 を別セクションでランキング**して上位3〜5件ずつ提案。各ポジションに**最終更新日と鮮度マークを併記**する：

```
## 🟢 即推薦可（N件）
1. {クライアント} / {ポジション} — 適合 X.X ／ 🟢 最終更新 YYYY-MM-DD
   - 推薦時に強調すべき点
   - 懸念点／事前に詰めるべきこと（🟡🔴 の場合は「推薦前に最新ステータス・枠の有無を要確認」を筆頭に）
   - 推薦の進め方（具体的なアクションリスト）

## 🟡 新規打診候補（N件）— P&L 未契約 or 未登録
1. {クライアント} / {ポジション} — 適合 X.X ／ 🟢 最終更新 YYYY-MM-DD / 一般公開URL: {url}
   - 提案：人事窓口にコンタクト推奨
   - 想定される取り組み（リテイナー打診 / 通常 Fee 提案 等）
```

### ポジション起点（`/match-position {社名} {ポジション}`）

1. 該当ポジション md（Craft）または SF Opportunity（注力 NO の場合）を fetch。**ポジションの最終更新日を確認し、🔴 要再同期なら先に SF/Notion を再取得してから評価する**
2. Craft の全候補者 md を一覧 → 各 md を fetch（規模大きければ active flag で絞る）
3. 各候補者について同5軸＋鮮度注記で評価、ランキング
