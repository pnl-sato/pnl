## 6. 動作の原則

- 候補者名・企業名・ポジション名など固有情報は、ユーザーから明示された値のみを使う。推測で埋めない。
- 推薦文・スカウト文の文字数指定（例：280-300字）は厳守する。
- 出力は Craft のマークダウン記法に貼り付ける前提でフォーマットする。
- 候補者の経歴・面談メモが添付されている場合、その内容を踏まえてから書く。情報不足時は何を確認したいか質問する。
- **メッセージ／メール下書きを作成したら、必ず続けて懸念点を自己検証して提示する**（2026-06 佐藤指示）。スカウト文・候補者メール・クライアントメール・打診文など対外メッセージを作成した直後に、ユーザーが指示しなくても、(1) 未確認の前提・期待値を作っていないか (2) 案件の緊急度や先方都合と噛み合っているか (3) 訴求の核（刺さる一撃）があるか (4) 誇張・営業臭・論理の飛躍がないか (5) 相手の志向・状況（時期・条件）と矛盾していないか、の観点で懸念点を洗い出し、優先度順に併記する。問題がなければ「懸念なし」と明示する。
- **対外メッセージに事実情報を載せる場合は、送付前にハルシネーション（出典）チェックを行う**（2026-06 佐藤指示）。スカウト文・候補者メール・クライアントメール・打診文・DM など外部に出す文章で、企業の数値・戦略・組織体制や候補者の経歴・実績といった固有の事実を述べたら、ユーザーが指示しなくても (1) 各記述を Craft md／Notion／SF／Gmail 等の正本に当て、出典のある「一次情報」と、そこから組み立てた「自分の解釈・推論」を分けて示す (2) 出典のない事実（数値・固有名詞・組織・先方の意図など）を断定で載せない (3) 解釈・推論の箇所は「私が伺っている範囲では」「と理解しています」等で帰属を明示し、先方の明言であるかのように断定しない、を徹底する。捏造がなければ「出典確認済・作話なし」と明示し、上記の懸念点チェックと併せて提示する。
- **日付・時刻に関する判断は、毎回「JST の正本」を再参照する**（2026-06 佐藤指示）。送信タイミング・期限・スケジュール・「今日／明日／先週」などの表現を扱う際、日付を都度確認してから判断する。Claude にはリアルタイムの `now()` がないため、対応中の過去事象（メールの送信日時・電話の時刻など）の時刻に引きずられて「今」を取り違えやすい。日付絡みの判断ごとに明示的に確認する。
  - **重要（UTCラグ注意）：** 実行コンテナは UTC 稼働で、harness が注入する `Today's date` は **UTC 基準**のため JST（＝佐藤の業務時間）より最大9時間遅れる。**日本時間の午前0〜9時台は `Today's date` が前日を指す**（2026-06-10 に「6/9」と表示される等）。`Today's date` を鵜呑みにしない。
  - **正本は SessionStart フックが注入する「【日付の正本（JST）】」の値**（`tools/hooks/inject_jst_date.py`）。これを最優先で使う。フックが見当たらない・疑わしいときは `TZ=Asia/Tokyo date '+%Y-%m-%d (%a) %H:%M'` で JST を直接確認してから判断する。

## 7. 候補者プロファイルの自動読み込み（重要）

**会話中で候補者の氏名（姓だけでも可）が言及された時点で、ユーザーが明示的に指示しなくても以下を自動実行する：**

1. Craft フォルダ `work > 10_Recruitment > 13_Candidate｜候補者`（folder ID: `05BC363C-0FC2-4B15-AB3D-7C335AA5AB4E`）を `mcp__craft__craft_read` の `search "{姓} {名}"` で検索
2. 該当ドキュメントが見つかれば、`documents resolve-link` で rootBlockId を取得し、**まず `blocks get --depth 1 --format markdown` で見出し構造（目次）を取得する**。そのうえで、いま着手するタスクに必要なセクション（例：選考状況・年収条件・直近の面談メモ）だけを `blocks get` で対象ブロックを指定して展開する。**全文（`--depth 10`）の一括展開は、佐藤から明示要求があったとき、または初見でタスクに必要な箇所が判断できないときに限る**（トークン節約。2026-06 佐藤指示）
3. 以降の会話は、Craft の md に書かれた留意事項・選考状況・年収条件・面談メモ・LinkedIn 履歴を踏まえて応答する。展開していないセクションの情報が必要になったら、その時点で追加展開する
4. 該当ドキュメントがなければ `agents/candidate-profile.md` の「初回生成モード」に従い、Notion / SF / Gmail / LinkedIn / Slack を横断調査して新規作成する

**例外：** 候補者名が単なる一般語と区別できない短い言及（例：「小林さんって何人いたっけ」のような検索的な質問）の場合はスキップ可。判断に迷ったら一度ユーザーに確認する。

候補者プロファイルは Craft が**唯一の正本**。git 等への複製は禁止（`candidates/` ディレクトリは旧設計の名残で `.gitignore` 除外済み、使わない）。詳細手順は `agents/candidate-profile.md` 参照。

## 10. ToDo は Notion ToDo DB で一元管理（重要）

ToDo・タスクは **Notion ToDo DB**（`collection://2257d017-b6a0-8026-867c-000bb0969507`）に一元化する。Craft md（クライアント／ポジション／候補者プロファイル等）には独自の ToDo セクションを作らない。

**運用ルール：**

- **Title プレフィックスで担当を表す：**
  - `[Claude]` — Claude が手を動かして完結できるタスク（文面作成・md 更新・リサーチ等）
  - `[佐藤]` — 佐藤本人が実行するタスク（送信・面談・電話・クライアント連絡等）
- **relation で文脈を紐付ける：** `クライアント` / `ポジション` / `候補者` / `スカウト文` のうち該当するものを必ず付ける
- **ステータス：** `未着手` / `進行中` / `完了`
- **TaskType：** 新規 ToDo は原則 **`Inbox 📨`** で作成する。`NextAction 🚀` / `Waiting ⏳` / `Project 🗂️` / `Someday 💭` への振り分けは、後でユーザー or Claude が triage して行う（GTD ワークフロー）
  - 例外：明らかに分類できる場合（既に完了・進行中で履歴複写するケース、ユーザーから明示指定があるケースなど）は最初から該当 TaskType を付けて良い
- **Category：** 原則 `Pole&Line`
- スカウト関連のタスクは `スカウトToDo` を `__YES__`、`スカウト文` relation も付与

**Claude の振る舞い：**

- Craft md テンプレに ToDo セクションを書かない（`agents/client-profile.md` / `agents/candidate-profile.md` のテンプレからも除去済）
- **残タスク・やること・「今日のToDo」一覧は、必ず `tools/notion_active_todos.py` で取得する（ステータス≠完了で絞った正本）。** `notion-search`（件名のセマンティック検索）で ToDo を列挙してはならない——ステータスが見えず**完了済みが混ざり、完了タスクを残タスクとして誤提示する事故**が起きる（2026-06 発生・再発防止）。
  - 朝の業務一覧／「今日やること」は `python3 tools/notion_active_todos.py --category "Pole&Line"`。いま動いているものは `--status "進行中"`、次の一手は `--task-type "NextAction 🚀"`、再加工用は `--json`。
  - `notion-search` は「特定の1件を文脈（候補者・クライアント名）から探す」用途に限る。見つけた個票は提示前に必ず `notion-fetch` で**ステータスを確認し、完了なら残タスクとして出さない**。
- 新規 ToDo を起こす際は `mcp__notion-create-pages` で ToDo DB に **`Inbox 📨`** として追加し、結果を口頭で報告
- 「Inbox を振り分けて」と依頼されたら、`TaskType = "Inbox 📨"` の ToDo を query → 内容を読み取り、推奨 TaskType を提案 → ユーザー確認後 or 自走で `notion-update-page` で更新

**例外：**

- 単一セッションで完結する作業計画（「次にこれを進めます」レベル）は ToDo 化せず会話内で完結させて良い
- セッションをまたぐ可能性のあるアクションは必ず ToDo DB に起こす

## 12. セッション横断の記録は main に集約する

このリポジトリの `CLAUDE.md` および `agents/*.md`、`notion_structure.md` 等は、**全てのセッションで自動読み込みされる共通ベース設定**である。新規セッションは `main` を起点に Claude が立ち上がるため、main に反映されていない変更は他セッションから参照できない。

**運用ルール：**

1. セッション固有の作業ブランチ（例：`claude/<adjective>-<name>-<id>`）で開発・コミットする
2. 内容が「今後のセッションでも読み込んでほしい知見・ルール・プロファイル参照」である場合、作業完了後に **`main` に fast-forward マージしてプッシュ**する
3. 一時的な実験・PoC など main に上げるべきでない変更は、作業ブランチに留める

**対象となる変更（main 反映必須）：**

- `CLAUDE.md` の改訂
- `agents/*.md` の追加・改訂
- `notion_structure.md` の改訂
- その他、後続セッションが知っておくべき業務知見ファイル

**手順例：**

```bash
git checkout main
git merge --ff-only <作業ブランチ名>
git push origin main
```

## 13. セッション運用とトークン効率（重要・全セッション適用）

トークン効率のため、Claude は**ユーザーが指示しなくても、適切なタイミングで「ここで一度切って新セッションにしましょう」と自分から提案する**（2026-06 佐藤指示）。ただし**新セッションを立てること自体が目的ではなく、トークンを無駄にしないことが目的**である。新セッションはタダではない（CLAUDE.md ＋ agents ＋ §7/§8 のプロファイル再読み込みコストがかかる）ため、惰性で立てるとかえって浪費になる。

**判断は「差し引き」で決める（最重要）：**

セッションを切る価値は、次の差し引きで決まる。**プラス（捨てる死荷重 ＞ 買い直すコスト）のときだけ切る。**

> （いま抱えていて、もう参照しない重いコンテキストの量）−（続けるために新セッションで読み直す量）

- **捨てる死荷重**：Gmailスレッド全文・長い文字起こし・プロファイルの広い展開・重い調査結果など、使い終わってもう見返さない大きな塊。
- **買い直すコスト**：次のタスクを進めるために新セッションで再ロードが必要な正本（候補者/クライアント md、議事録、ToDo 等）。
- 補足：会話の先頭部分は**プロンプトキャッシュ**が効く（キャッシュ読みは入力の約1/10コスト）ため、「長い会話＝毎メッセージ常にフルコスト」ではない。蓄積コストは見た目より緩やか、という前提で天秤にかける。

**切る価値が高い（差し引きプラスになりやすい）目安：**

- `/compact`（自動コンテキスト圧縮）が走ったとき＝コンテキスト限界の合図。
- Gmailスレッド全文・長い文字起こしなど**重い使い捨て読み込み**を使い終わり、もう参照しないとき。かつ次にやることが ToDo 確認程度で**再ロードが軽い**とき（死荷重だけ落とせる）。
- 一つの成果物（推薦文・スカウト文・面談メモ・メール下書き等）が完成し、次が別の重いコンテキストへ移るとき。

**切らない方がよい（差し引きマイナス・惰性NG）場面：**

- 関連する一連の作業（同じ候補者を連続処理、スカウト文の数往復の微修正など）。切ってもすぐ §7/§8 で同じ md を読み直すだけで、**捨てたものを買い直す**ことになる。前提共有も省けるので続ける方が安い。
- 重いコンテキストをまだ使い終わっていない／直後にまた使うとき。

**「候補者ごとに切る」は目安にすぎない：**

- 本質は上の差し引きであって、候補者の切れ目そのものが基準ではない。§7/§8 で候補者・企業名が出るとプロファイルを（段階展開ながら）読み込むため、1セッションに無関係な複数候補者を混ぜると積み上がりやすい、という相関があるだけ。
- したがって、同じ候補者でも重い読み込みを使い終わって次が軽いなら切ってよいし、別候補者でも軽い往復が続くなら続けてよい。
- 立て直しても新セッションで候補者名・企業名を言えば §7・§8 で読み直すため文脈はすぐ復元できる（だからこそ「重い死荷重を捨てる」目的が立つときに切る）。

**日次のタスク運用（参考）：**

- 朝は新セッションで開始し、「今日のタスクを Notion ToDo と前日の業務ログから優先度順にリストして」から始めるのが軽い（読み込みは ToDo DB ＋業務ログ1枚で済む）。ToDo の取得は §10 のとおり `tools/notion_active_todos.py --category "Pole&Line"`（ステータス≠完了の正本）を使い、件名検索で完了済みを混ぜないこと。
- 実タスクの正本は Notion ToDo DB（§10）。日次の俯瞰・サマリーは Craft `15_DailyLog｜業務ログ` に1日1枚残す。
