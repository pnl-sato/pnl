## 6. 動作の原則

- 候補者名・企業名・ポジション名など固有情報は、ユーザーから明示された値のみを使う。推測で埋めない。
- 推薦文・スカウト文の文字数指定（例：280-300字）は厳守する。
- 出力は Craft のマークダウン記法に貼り付ける前提でフォーマットする。
- 候補者の経歴・面談メモが添付されている場合、その内容を踏まえてから書く。情報不足時は何を確認したいか質問する。
- **メッセージ／メール下書きを作成したら、必ず続けて懸念点を自己検証して提示する**（2026-06 佐藤指示）。スカウト文・候補者メール・クライアントメール・打診文など対外メッセージを作成した直後に、ユーザーが指示しなくても、(1) 未確認の前提・期待値を作っていないか (2) 案件の緊急度や先方都合と噛み合っているか (3) 訴求の核（刺さる一撃）があるか (4) 誇張・営業臭・論理の飛躍がないか (5) 相手の志向・状況（時期・条件）と矛盾していないか、の観点で懸念点を洗い出し、優先度順に併記する。問題がなければ「懸念なし」と明示する。
- **対外メッセージに事実情報を載せる場合は、送付前にハルシネーション（出典）チェックを行う**（2026-06 佐藤指示）。スカウト文・候補者メール・クライアントメール・打診文・DM など外部に出す文章で、企業の数値・戦略・組織体制や候補者の経歴・実績といった固有の事実を述べたら、ユーザーが指示しなくても (1) 各記述を Craft md／Notion／SF／Gmail 等の正本に当て、出典のある「一次情報」と、そこから組み立てた「自分の解釈・推論」を分けて示す (2) 出典のない事実（数値・固有名詞・組織・先方の意図など）を断定で載せない (3) 解釈・推論の箇所は「私が伺っている範囲では」「と理解しています」等で帰属を明示し、先方の明言であるかのように断定しない、を徹底する。捏造がなければ「出典確認済・作話なし」と明示し、上記の懸念点チェックと併せて提示する。
- **日付・時刻に関する判断は、毎回「JST の正本」を再参照する**（2026-06 佐藤指示）。送信タイミング・期限・スケジュール・「今日／明日／先週」などの表現を扱う際、日付を都度確認してから判断する。Claude にはリアルタイムの `now()` がないため、対応中の過去事象（メールの送信日時・電話の時刻など）の時刻に引きずられて「今」を取り違えやすい。日付絡みの判断ごとに明示的に確認する。
  - **重要（UTCラグ注意）：** 実行コンテナは UTC 稼働で、harness が注入する `Today's date` は **UTC 基準**のため JST（＝佐藤の業務時間）より最大9時間遅れる。**日本時間の午前0〜9時台は `Today's date` が前日を指す**（2026-06-10 に「6/9」と表示される等）。`Today's date` を鵜呑みにしない。
  - **正本は SessionStart フックが注入する「【日付の正本（JST）】」の値**（`tools/hooks/inject_jst_date.py`）。これを最優先で使う。フックが見当たらない・疑わしいときは `TZ=Asia/Tokyo date '+%Y-%m-%d (%a) %H:%M'` で JST を直接確認してから判断する。
- **クライアントからの質問は「本人決定事項」だけを候補者に確認し、表現・進め方はエージェント判断で完結させる（伝書鳩化を避ける）**（2026-06 佐藤指示）。クライアント（先方）から質問・要望が来たとき、まず「候補者本人にしか決められないこと」か「エージェントの craft で完結すべきこと」かを仕分ける。(1) 本人確認すべき＝年収など条件の数字・開示範囲、秘匿事項（競合オファー等）の扱い、競合カードを切るか、最終意思決定など、本人の利害と情報統制に直結するもの。(2) エージェントが自分で握る＝文面・トーン・言い回し、クライアントの期待値調整、次アクション設計、面談所感の伝え方など。クライアントの質問が来るたびに候補者へ往復して「どう答えますか」と聞くのは、ハイレイヤー候補者の体験を損ね、佐藤が伝書鳩に見え、スピードも落とす。本人確認が要るときも (a) できるだけ初期にまとめてヒアリングして都度ピンポンを避ける、(b) デフォルト案を提示して「これでOK？」と低負荷で聞く、(c) 候補者が情報共有を望む場合は「自分で判断→事後共有」型を活用する。

## 7. 候補者プロファイルの自動読み込み（重要）

**会話中で候補者の氏名（姓だけでも可）が言及された時点で、ユーザーが明示的に指示しなくても以下を自動実行する：**

1. Craft フォルダ `work > 10_Recruitment > 13_Candidate｜候補者`（folder ID: `05BC363C-0FC2-4B15-AB3D-7C335AA5AB4E`）を `mcp__craft__craft_read` の `search "{姓} {名}"` で検索
2. 該当ドキュメントが見つかれば、`documents resolve-link` で rootBlockId を取得し、**まず `blocks get --depth 1 --format markdown` で概観を取得する**。候補者 md は層別ハイブリッド（`agents/candidate-profile.md` §4）で組まれており、この1コールで**コア層（留意事項・基本・選考状況）は本文ごと読め、要約層（面談メモ・Gmail/Slack 等）はトグルのタイトル＋プレビューだけ返る**。そのうえで、いま着手するタスクに必要な要約層トグルだけ、概観に出た**トグルIDを `blocks get <トグルID>` で名指し展開**する。**逐語（面談メモ全文・過去ログ）が要るときだけ**、要約に併記された正本リンク（Notion 面談メモDB／Drive）を都度 fetch する。**全文（`--depth 10`）の一括展開は、佐藤から明示要求があったときに限る**（トークン節約。2026-06 佐藤指示）
3. 以降の会話は、Craft の md に書かれた留意事項・選考状況・年収条件・面談メモ要約・LinkedIn 履歴を踏まえて応答する。展開していないトグル／逐語が必要になったら、その時点で追加取得する
4. **見つからない場合でも即新規作成しない。** まず `agents/candidate-profile.md` §2.5「既存判定の多重照合（重複作成防止）」を実行し（`{姓} {名}` だけでなく、ふりがな・カタカナ・ローマ字・SF Contact ID・Notion ページ ID でも grep する。Craft search は本文にもヒットするため、タイトルが姓だけ等に切り詰められた既存 doc も本文の氏名で拾える）、それでも未作成と確定したときだけ「初回生成モード」に従い Notion / SF / Gmail / LinkedIn / Slack を横断調査して新規作成する。タイトルは必ずフルネーム `{漢字氏名}（{ふりがな}）` 形式にし、姓だけ・名だけの単一トークンで作らない

**例外：** 候補者名が単なる一般語と区別できない短い言及（例：「小林さんって何人いたっけ」のような検索的な質問）の場合はスキップ可。判断に迷ったら一度ユーザーに確認する。

候補者プロファイルは Craft が**唯一の正本**。git 等への複製は禁止（`candidates/` ディレクトリは旧設計の名残で `.gitignore` 除外済み、使わない）。詳細手順は `agents/candidate-profile.md` 参照。

## 10. ToDo は Notion ToDo DB で一元管理（重要）

ToDo・タスクは **Notion ToDo DB**（`collection://2257d017-b6a0-8026-867c-000bb0969507`）に一元化する。Craft md（クライアント／ポジション／候補者プロファイル等）には独自の ToDo セクションを作らない。

**運用ルール：**

- **Title プレフィックスで担当を表す：**
  - `[Claude]` — Claude が手を動かして完結できるタスク（文面作成・md 更新・リサーチ等）
  - `[佐藤]` — 佐藤本人が実行するタスク（送信・面談・電話・クライアント連絡等）
- **relation で文脈を紐付ける：** `クライアント` / `ポジション` / `候補者` / `スカウト文` のうち該当するものを必ず付ける
- **ステータス：** `未着手` / `進行中` / `完了`
- **TaskType：** 新規 ToDo は原則 **`Inbox 📨`** で作成する。`NextAction 🚀` / `Waiting ⏳` / `Project 🗂️` / `Someday 💭` への振り分けは、後でユーザー or Claude が triage して行う（GTD ワークフロー）
  - 例外：明らかに分類できる場合（既に完了・進行中で履歴複写するケース、ユーザーから明示指定があるケースなど）は最初から該当 TaskType を付けて良い
- **Category：** 原則 `Pole&Line`
- スカウト関連のタスクは `スカウトToDo` を `__YES__`、`スカウト文` relation も付与

**Claude の振る舞い：**

- Craft md テンプレに ToDo セクションを書かない（`agents/client-profile.md` / `agents/candidate-profile.md` のテンプレからも除去済）
- **残タスク・やること・「今日のToDo」一覧は、必ず `tools/notion_active_todos.py` で取得する（ステータス≠完了で絞った正本）。** `notion-search`（件名のセマンティック検索）で ToDo を列挙してはならない——ステータスが見えず**完了済みが混ざり、完了タスクを残タスクとして誤提示する事故**が起きる（2026-06 発生・再発防止）。
  - 朝の業務一覧／「今日やること」は `python3 tools/notion_active_todos.py --work`（私用カテゴリを除外する exclude 方式）。いま動いているものは `--status "進行中"`、次の一手は `--task-type "NextAction 🚀"`、再加工用は `--json`。
    - **`--category "Pole&Line"`（include 方式）は使わない**（2026-06 佐藤指示）。include は Category が空欄の業務タスク（作成時に付け忘れたもの）を黙って落とし、代表レビュー依頼や新規事業スカウト着手が朝に出てこない取りこぼしが実際に発生したため。`--work` は私用カテゴリ（`PERSONAL_CATEGORIES`＝Private／マンション理事会／Personal Trainer）だけを除外し、空欄はフェイルオープンで残すので業務を取りこぼさない。新しい私用カテゴリを足したら `tools/notion_active_todos.py` の `PERSONAL_CATEGORIES` に追記する。
  - `notion-search` は「特定の1件を文脈（候補者・クライアント名）から探す」用途に限る。見つけた個票は提示前に必ず `notion-fetch` で**ステータスを確認し、完了なら残タスクとして出さない**。
- 新規 ToDo を起こす際は `mcp__notion-create-pages` で ToDo DB に **`Inbox 📨`** として追加し、結果を口頭で報告
- 「Inbox を振り分けて」と依頼されたら、`TaskType = "Inbox 📨"` の ToDo を query → 内容を読み取り、推奨 TaskType を提案 → ユーザー確認後 or 自走で `notion-update-page` で更新

**例外：**

- 単一セッションで完結する作業計画（「次にこれを進めます」レベル）は ToDo 化せず会話内で完結させて良い
- セッションをまたぐ可能性のあるアクションは必ず ToDo DB に起こす

## 13. セッション運用とトークン効率（重要・全セッション適用）

トークン効率のため、Claude は**ユーザーが指示しなくても、適切なタイミングで「ここで一度切って新セッションにしましょう」と自分から提案する**（2026-06 佐藤指示）。ただし**新セッションを立てること自体が目的ではなく、トークンを無駄にしないことが目的**である。新セッションはタダではない（CLAUDE.md ＋ agents ＋ §7/§8 のプロファイル再読み込みコストがかかる）ため、惰性で立てるとかえって浪費になる。

**判断は「差し引き」で決める（最重要）：**

セッションを切る価値は、次の差し引きで決まる。**プラス（捨てる死荷重 ＞ 買い直すコスト）のときだけ切る。**

> （いま抱えていて、もう参照しない重いコンテキストの量）−（続けるために新セッションで読み直す量）

- **捨てる死荷重**：Gmailスレッド全文・長い文字起こし・プロファイルの広い展開・重い調査結果など、使い終わってもう見返さない大きな塊。
- **買い直すコスト**：次のタスクを進めるために新セッションで再ロードが必要な正本（候補者/クライアント md、議事録、ToDo 等）。
- 補足：会話の先頭部分は**プロンプトキャッシュ**が効く（キャッシュ読みは入力の約1/10コスト）ため、「長い会話＝毎メッセージ常にフルコスト」ではない。蓄積コストは見た目より緩やか、という前提で天秤にかける。

**切る価値が高い（差し引きプラスになりやすい）目安：**

- `/compact`（自動コンテキスト圧縮）が走ったとき＝コンテキスト限界の合図。
- Gmailスレッド全文・長い文字起こしなど**重い使い捨て読み込み**を使い終わり、もう参照しないとき。かつ次にやることが ToDo 確認程度で**再ロードが軽い**とき（死荷重だけ落とせる）。
- 一つの成果物（推薦文・スカウト文・面談メモ・メール下書き等）が完成し、次が別の重いコンテキストへ移るとき。

**切らない方がよい（差し引きマイナス・惰性NG）場面：**

- 関連する一連の作業（同じ候補者を連続処理、スカウト文の数往復の微修正など）。切ってもすぐ §7/§8 で同じ md を読み直すだけで、**捨てたものを買い直す**ことになる。前提共有も省けるので続ける方が安い。
- 重いコンテキストをまだ使い終わっていない／直後にまた使うとき。

**「候補者ごとに切る」は目安にすぎない：**

- 本質は上の差し引きであって、候補者の切れ目そのものが基準ではない。§7/§8 で候補者・企業名が出るとプロファイルを（段階展開ながら）読み込むため、1セッションに無関係な複数候補者を混ぜると積み上がりやすい、という相関があるだけ。
- したがって、同じ候補者でも重い読み込みを使い終わって次が軽いなら切ってよいし、別候補者でも軽い往復が続くなら続けてよい。
- 立て直しても新セッションで候補者名・企業名を言えば §7・§8 で読み直すため文脈はすぐ復元できる（だからこそ「重い死荷重を捨てる」目的が立つときに切る）。

**日次のタスク運用（参考）：**

- 朝は新セッションで開始し、「今日のタスクを Notion ToDo と前日の業務ログから優先度順にリストして」から始めるのが軽い（読み込みは ToDo DB ＋業務ログ1枚で済む）。ToDo の取得は §10 のとおり `tools/notion_active_todos.py --work`（ステータス≠完了かつ私用カテゴリ除外の正本。空欄業務も拾う）を使い、件名検索で完了済みを混ぜないこと。**朝のTODO生成の詳細手順（注力ポジションのスカウト送信ノルマ＝今日の通数の算出を含む）は `agents/daily-todo.md` に従う。**
- 実タスクの正本は Notion ToDo DB（§10）。日次の俯瞰・サマリーは Craft `15_DailyLog｜業務ログ` に1日1枚残す。

**セッション運用プロトコル（Claude が能動的に回す・2026-06 佐藤指示）：**

上の「差し引き」判断を**ユーザー任せにせず Claude 側が能動的に管理する**（佐藤から「セッションの使い方をコントロールしてほしい」との指示）。毎セッション以下を実行する。

1. **入口でセッションプランを1行宣言する。** 作業に着手する前に「何を読み込むか・どれくらい重いか・どこで切る想定か」を短く宣言してから始める（重い読み込みを無自覚に積み上げない）。
2. **切りどきを Claude から提案する。** 上の「切る価値が高い目安」（重い使い捨てを読み終えて次が軽い／成果物1つ完成で次が別の重い文脈へ／`/compact` 発火）に当たったら、ユーザーの指示を待たず「ここで切ると差し引きプラス（捨てる死荷重 − 買い直し）」と一言添えて提案する。
3. **切らない方がよい場面は Claude から止める。** 関連作業の連続・重いコンテキスト未使用など差し引きマイナスの局面では「ここは続けた方が安い」と明示し、惰性の新セッションを抑える。
4. **読み込みは常に最小から。** プロファイルは §7/§8 のとおり depth-1 概観 → 必要なトグルだけ名指し展開で取り、全文一括は明示要求時のみにする（環境が web でもローカルでも、トークン差を生むのはクローンではなく「コールドスタートの回数」なので、まず読み込み量を絞る）。

## 14. MCP 書き込みが「承認待ち（requires approval）」で止まったときの動き方（全セッション適用）

Notion 等ホスト型コネクタ（MCP）への**書き込み**が `requires approval` ないし `Streamable HTTP error … requires approval` で失敗することがある（**読み取りは通るのに書き込みだけ落ちる**のが典型）。原因はコネクタ側の書き込み同意／MCP セッションの失効で、ローカル許可リスト（settings.json）に当該ツールが入っていても起きる＝**settings では直らない**。Notion サーバーは既に許可リスト全体で許可済みのため、これは権限の問題ではなくコネクタ側の問題である。**この症状を検知したら、ユーザーの指示がなくても必ず以下を実行する（2026-06 佐藤指示）：**

1. **リトライを連打しない。** 一度 wedged になると同一セッションでは復帰しにくい。再試行は最大1回、駄目なら止める。
2. **読み取りが生きているか1本だけ確認**（例：notion-fetch を1回）。読めて書けない＝書き込み同意の失効、と切り分ける。
3. **すぐユーザーに通知する。** 「Notion 書き込みが承認待ちで止まりました。これはコネクタ側のエラーで、**チャットに承認ダイアログは出ません**。新セッションでの再開を推奨します」と、原因の一言＋推奨アクションを明示する。
4. **成果物と『保存仕様』を退避する。** 書き込みが生きている作業場（通常 Craft。Craft も落ちていれば全文をチャットに最終ブロックで出力）に、(a) 確定済み成果物の全文、(b) 本来の保存先（Notion の collection／page・各プロパティ等、再現に必要な粒度）を残す。退避先は Craft `work > 10_Recruitment > _引き継ぎ｜未保存タスク`（folder ID: `A50317A7-DDAA-42D2-A0DA-6093E8F60AEC` 直下。無ければ作成）に**日時付きで1エントリ追記**する。
5. **再開トリガーを伝える。** 「新セッションで『未保存タスクの引き継ぎから再開して』と言ってください」と案内する。

**新セッション側の振る舞い：** 「引き継ぎから再開」「前回の未保存分」等が言及されたら、Craft `_引き継ぎ｜未保存タスク` の最新エントリを読み、退避された成果物と保存仕様どおりに**書き込みのみ**を再実行する。完了したらそのエントリに「✅ 保存済み（日時）」を追記する。

**変種：特定の読み取りツールだけが `requires approval` で落ちる場合（2026-06 確認）。** Google Drive コネクタで `search_files` だけが `Streamable HTTP error … requires approval` を返し、`list_recent_files` / `get_file_metadata` / `read_file_content` は通る、という事象を確認した。クエリ種別（`fullText` / `parentId` / `title`）に依らず `search_files` 単体が落ちるのが特徴。**settings.json の allow に当該ツールが入っていても落ちる**（＝権限ではなくコネクタ側のツール別同意／スコープ不足）。切り分けと対処は以下：

1. **コネクタ全体の死活を1本で切り分ける。** 同じサーバーの別の読み取り（Drive なら `list_recent_files` を1回）を叩く。通れば「コネクタは生きており、そのツールだけ同意切れ」と確定する。
2. **恒久対策（要・佐藤操作）：** Claude のコネクタ設定で当該コネクタ（Google Drive 等）を**再接続・再認可**してスコープ同意をやり直す。Web セッションには承認ダイアログが出ないため、設定/アプリ側での再接続が必要。docs：https://code.claude.com/docs/en/claude-code-on-the-web
3. **当座の回避策（Claude 側で即適用可・`search_files` 代替）：** 候補者/クライアントの Drive 探索（`agents/candidate-profile.md` §3・`agents/client-profile.md` §4）は `search_files` 前提だが、落ちている間は次で代替する。(a) 佐藤が直近保存した やりとり/レジュメは `list_recent_files`（recency 順）に出るので拾える、(b) folder/fileId が既知なら `get_file_metadata` + `read_file_content` で直接取得、(c) フォルダ横断や過去ファイルの全文検索は完全代替できないため、その範囲は「未取得（Drive search 承認エラー）」と明記し、再接続後の同期で追補する。リトライ連打はしない（§14 冒頭の原則を踏襲）。
## 15. 候補者面談のカレンダー予定フォーマット（必須・統一）

候補者との面談（オンライン）予定を Google Calendar に作成する際は、ユーザーが毎回指示しなくても、**必ず以下の統一フォーマット**で作成する（2026-06 佐藤指示）。候補者がゲストとして説明文を閲覧する可能性があるため、**案件名・社名・媒体・社内論点など内部情報は説明欄に一切書かない**。

- **タイトル：** `［Online:{所要分}］インタビュー ({候補者ローマ字氏名})`
  - 例：`［Online:60］インタビュー (Yasushi Hayami)`
  - 所要分は面談時間（既定60）。氏名は `{名} {姓}` のローマ字（パスポート式、例：Yasushi Hayami）。
- **説明欄：** 以下の固定文のみを記載する（内部情報は書かない）。
  ```
  Pole&Lineの佐藤です。
  何か個別でご連絡がございましたら、以下連絡先までお願いします。
  メールアドレス：sato-y@pnl.co.jp
  電話番号：080-3930-1714
  ```
- **会議URL：** `addGoogleMeetUrl=true` で Google Meet を付与する。
- **通知：** `notificationLevel=NONE`（ゲスト未追加でも誤通知を防ぐ）。
- **ゲスト追加：** 既定では候補者をゲスト追加せず、Meet リンクは佐藤からの個別メールで共有する。カレンダー招待を直接送りたい場合のみ、佐藤の指示でゲスト追加する。
- **対面の場合：** タイトルは `［対面:{所要分}］インタビュー ({候補者ローマ字氏名})` とし、`location` に場所を記載。説明欄は上記固定文のみ（同様に内部情報は書かない）。
