## 6. 動作の原則

- 候補者名・企業名・ポジション名など固有情報は、ユーザーから明示された値のみを使う。推測で埋めない。
- 推薦文・スカウト文の文字数指定（例：280-300字）は厳守する。
- 出力は Craft のマークダウン記法に貼り付ける前提でフォーマットする。
- 候補者の経歴・面談メモが添付されている場合、その内容を踏まえてから書く。情報不足時は何を確認したいか質問する。
- **メッセージ／メール下書きを作成したら、必ず続けて懸念点を自己検証して提示する**（2026-06 佐藤指示）。スカウト文・候補者メール・クライアントメール・打診文など対外メッセージを作成した直後に、ユーザーが指示しなくても、(1) 未確認の前提・期待値を作っていないか (2) 案件の緊急度や先方都合と噛み合っているか (3) 訴求の核（刺さる一撃）があるか (4) 誇張・営業臭・論理の飛躍がないか (5) 相手の志向・状況（時期・条件）と矛盾していないか、の観点で懸念点を洗い出し、優先度順に併記する。問題がなければ「懸念なし」と明示する。
- **日付・時刻に関する判断は、毎回 `Today's date` を再参照する**（2026-06 佐藤指示）。送信タイミング・期限・スケジュール・「今日／明日／先週」などの表現を扱う際、セッション冒頭の context にある `Today's date` を都度確認してから判断する。Claude にはリアルタイムの `now()` がないため、対応中の過去事象（メールの送信日時・電話の時刻など）の時刻に引きずられて「今」を取り違えやすい。日付絡みの判断ごとに明示的に確認する。

**会話中で候補者の氏名（姓だけでも可）が言及された時点で、ユーザーが明示的に指示しなくても以下を自動実行する：**

1. Craft フォルダ `work > 10_Recruitment > 13_Candidate｜候補者`（folder ID: `05BC363C-0FC2-4B15-AB3D-7C335AA5AB4E`）を `mcp__craft__craft_read` の `search "{姓} {名}"` で検索

**トリガーキーワード：** 「1on1」「上長」「代表」「議事録」「先週の方針」「面談スタンス」「方針確認」「上から言われた」など。

## 10. ToDo は Notion ToDo DB で一元管理（重要）

ToDo・タスクは **Notion ToDo DB**（`collection://2257d017-b6a0-8026-867c-000bb0969507`）に一元化する。Craft md（クライアント／ポジション／候補者プロファイル等）には独自の ToDo セクションを作らない。

**運用ルール：**

- **Title プレフィックスで担当を表す：**
  - `[Claude]` — Claude が手を動かして完結できるタスク（文面作成・md 更新・リサーチ等）
  - `[佐藤]` — 佐藤本人が実行するタスク（送信・面談・電話・クライアント連絡等）
- **relation で文脈を紐付ける：** `クライアント` / `ポジション` / `候補者` / `スカウト文` のうち該当するものを必ず付ける
- **ステータス：** `未着手` / `進行中` / `完了`
- **TaskType：** 新規 ToDo は原則 **`Inbox 📨`** で作成する。`NextAction 🚀` / `Waiting ⏳` / `Project 🗂️` / `Someday 💭` への振り分けは、後でユーザー or Claude が triage して行う（GTD ワークフロー）
  - 例外：明らかに分類できる場合（既に完了・進行中で履歴複写するケース、ユーザーから明示指定があるケースなど）は最初から該当 TaskType を付けて良い
- **Category：** 原則 `Pole&Line`
- スカウト関連のタスクは `スカウトToDo` を `__YES__`、`スカウト文` relation も付与

**Claude の振る舞い：**

- Craft md テンプレに ToDo セクションを書かない（`agents/client-profile.md` / `agents/candidate-profile.md` のテンプレからも除去済）
- 「○○ の ToDo は？」と聞かれたら Notion ToDo DB を `mcp__notion-search` または `mcp__notion-fetch` で query（relation で絞り込み）
- 新規 ToDo を起こす際は `mcp__notion-create-pages` で ToDo DB に **`Inbox 📨`** として追加し、結果を口頭で報告
- 「Inbox を振り分けて」と依頼されたら、`TaskType = "Inbox 📨"` の ToDo を query → 内容を読み取り、推奨 TaskType を提案 → ユーザー確認後 or 自走で `notion-update-page` で更新

**例外：**

- 単一セッションで完結する作業計画（「次にこれを進めます」レベル）は ToDo 化せず会話内で完結させて良い
- セッションをまたぐ可能性のあるアクションは必ず ToDo DB に起こす

## 11. 転職DB検索フィルタ作成時のマスター参照（重要）

ビズリーチ等の転職DB（スカウト媒体）でターゲットを絞り込む**検索フィルタ（検索条件）**を作成・調整する際は、各DBが持つ選択肢の正規一覧（職種・業種・年収・学歴 など）を Craft に **DBごと**にまとめてある。これを正本とし、実在しない選択肢を推測で作らず、必ずこのマスターに存在する選択肢でフィルタを組む。

**保存先：** Craft `work > 10_Recruitment > スカウト媒体`（folder ID: `E87A0808-42C9-4304-AB9E-32A39A50F2F7`）配下に、**DBごとのサブフォルダ**として整理する。

- **ビズリーチ**（folder ID: `819033D3-ADB1-493E-AD1D-6B2123A7025A`。現状はこのDBのみ整備済み）
  - `ビズリーチ職種カテゴリ・マスターデータ`
  - `ビズリーチ業種カテゴリ・マスターデータ`
  - `ビズリーチ年収条件・マスターデータ`
  - `ビズリーチ学歴条件・マスターデータ`
  - `ビズリーチ英語力条件・マスターデータ`
  - `ビズリーチ・マネジメント経験条件・マスターデータ`
  - `ビズリーチ登録社数条件・マスターデータ`

**実行手順：**

1. 検索フィルタ作成・調整の依頼を受けたら、対象DBを確認する（指定がなければビズリーチ）。
2. 対象DBのサブフォルダを `mcp__craft__craft_read` の `documents list --folder <サブフォルダID>` で開き、必要な条件のマスター doc を `blocks get --format markdown` で展開する。
3. マスターに存在する選択肢のみを使ってフィルタを組む。マスターにない条件・選択肢が必要になった場合は、推測で埋めず**ユーザーに確認**する。
4. 新しいDB（LinkedIn・dodaX 等）のマスターが追加された場合も、同じ `スカウト媒体/{DB名}/` 配下に置かれる前提で同様に参照する。

**例外：** 単に「どんな条件があったっけ？」程度の参照確認はマスターを開いて答えれば足りる。フィルタそのものを設計するタスクでは必ず該当マスターを展開してから組む。

## 12. セッション横断の記録は main に集約する

このリポジトリの `CLAUDE.md` および `agents/*.md`、`notion_structure.md` 等は、**全てのセッションで自動読み込みされる共通ベース設定**である。新規セッションは `main` を起点に Claude が立ち上がるため、main に反映されていない変更は他セッションから参照できない。

**運用ルール：**

1. セッション固有の作業ブランチ（例：`claude/<adjective>-<name>-<id>`）で開発・コミットする
2. 内容が「今後のセッションでも読み込んでほしい知見・ルール・プロファイル参照」である場合、作業完了後に **`main` に fast-forward マージしてプッシュ**する
3. 一時的な実験・PoC など main に上げるべきでない変更は、作業ブランチに留める

**対象となる変更（main 反映必須）：**

- `CLAUDE.md` の改訂
- `agents/*.md` の追加・改訂
- `notion_structure.md` の改訂
- その他、後続セッションが知っておくべき業務知見ファイル

**手順例：**

```bash
git checkout main
git merge --ff-only <作業ブランチ名>
git push origin main
```
